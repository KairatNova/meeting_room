## Введение

1. **Актуальность.**  
   В организациях с несколькими переговорными комнатами ручной учёт бронирований (таблицы Excel, бумажные журналы) приводит к двойным бронированиям, потере рабочего времени и организационным конфликтам. Существующие коммерческие системы (Robin, Skedda, Microsoft Outlook Room Booking) требуют платных корпоративных лицензий и не имеют интеграции с Telegram — мессенджером, широко используемым в СНГ. Разработка собственного веб-приложения с Telegram-верификацией и гибкими бизнес-правилами является актуальной практической задачей.

2. **Цель.**  
   Разработать полнофункциональное веб-приложение для автоматизации бронирования переговорных комнат с ролевым разграничением доступа и верификацией пользователей через Telegram-бота.

3. **Задачи:**
   - Проанализировать предметную область и изучить существующие аналоги.
   - Сформулировать функциональные и нефункциональные требования к системе.
   - Спроектировать архитектуру приложения и схему реляционной базы данных.
   - Разработать серверную часть (REST API) на базе FastAPI и PostgreSQL.
   - Разработать клиентскую часть (SPA) на базе React и TypeScript.
   - Реализовать бизнес-логику: проверку конфликтов бронирований, буфер времени, политику отмены.
   - Интегрировать верификацию аккаунтов через Telegram Bot API.
   - Разработать автоматические тесты ключевых модулей (Pytest, Vitest).
   - Выполнить развёртывание приложения на облачной платформе (Render).

4. **Объект исследования.**  
   Процесс управления бронированием переговорных комнат в организации.

5. **Предмет исследования.**  
   Методы и технологии разработки веб-приложений для автоматизации управления ресурсами организации.

6. **Методы:**  
   Сравнительный анализ аналогов, объектно-ориентированное проектирование, реляционное моделирование данных, REST-архитектурный стиль, итеративная разработка, модульное и интеграционное тестирование.

7. **Практическая значимость.**  
   Разработанная система может быть внедрена в учебное заведение, офисный центр или коворкинг для автоматизации процесса бронирования. Приложение задеплоено на облачной платформе и готово к эксплуатации.

## Глава 1. Анализ предметной области и выбор технологий

**1.1. Описание предметной области и проблематика**
- Роль переговорных комнат в организации.
- Участники процесса: пользователи и администраторы.
- Проблемы ручного управления: двойные бронирования, отсутствие прозрачности, нет контроля отмен.
- Ключевые сущности: Пользователь, Комната, Бронирование, Код верификации.

**1.2. Обзор и сравнительный анализ существующих аналогов**
- Анализ систем: Google Calendar Resources, Microsoft Outlook Room Booking, Robin Powered, Skedda.
- Сравнительная таблица: стоимость, русский язык, Telegram-интеграция, роли, API, визуальный календарь.
- Вывод о необходимости разработки собственного решения.

**1.3. Постановка задачи и требования к системе**
- Функциональные требования (ФТ):
  - ФТ-1: Управление пользователями (регистрация, верификация через Telegram, вход, профиль).
  - ФТ-2: Управление комнатами (CRUD только для администраторов, фотографии).
  - ФТ-3: Управление бронированиями (создание, просмотр, отмена, детали).
  - ФТ-4: Управление администраторами (назначение и снятие роли).
  - ФТ-5: Бизнес-правила (буфер 15 мин, длительность 30 мин – 6 часов, дедлайн отмены 30 мин).
- Нефункциональные требования: безопасность (JWT, bcrypt), адаптивность, документированность (Swagger), тестируемость.
- Варианты использования (Use Cases).

**1.4. Выбор и обоснование технологического стека**
- Backend: FastAPI (Python) — сравнение с Django REST Framework, Express.js.
- База данных: PostgreSQL — сравнение с MySQL, SQLite; ORM SQLAlchemy, миграции Alembic.
- Frontend: React + TypeScript + Vite — сравнение с Vue, Angular; стилизация Tailwind CSS.
- Дополнительно: FullCalendar (календарь), JWT (авторизация), Telegram Bot API (верификация).
- Сводная таблица стека с обоснованием каждого выбора.

## Глава 2. Проектирование информационной системы

**2.1. Архитектура приложения**
- Трёхуровневая клиент-серверная архитектура: Presentation / Application / Data Layer.
- Схема взаимодействия компонентов: React SPA ↔ FastAPI ↔ PostgreSQL, интеграция Telegram.
- Принципы организации backend: разделение на слои api / models / schemas / services.

**2.2. Проектирование базы данных**
- Сущности и атрибуты:
  - `users` (id, name, email, hashed_password, telegram_username, phone, telegram_chat_id, is_verified, is_admin, created_at).
  - `rooms` (id, name, capacity, location, description, amenities, is_active).
  - `room_photos` (id, room_id → rooms, url).
  - `bookings` (id, room_id → rooms, user_id → users, start_time, end_time, status, created_at).
  - `email_verification_codes` (id, user_id → users, code, expires_at).
- Связи между таблицами (1:N, CASCADE).
- ER-диаграмма.

**2.3. Проектирование REST API**
- Принципы REST: ресурсы, методы (GET/POST/PATCH/DELETE), статус-коды.
- Группы эндпоинтов: `/api/auth`, `/api/rooms`, `/api/bookings`, `/api/admin`.
- Сводная таблица эндпоинтов с методами, путями и уровнями доступа.
- Swagger/OpenAPI-документация.

**2.4. Проектирование пользовательского интерфейса**
- Карта экранов и маршрутизация (React Router v6).
- Защищённые маршруты: PrivateRoute (авторизация), AdminRoute (роль admin).
- Описание ключевых экранов: дашборд, страница комнаты, мои бронирования, детали брони, админ-панель.
- UX-решения: SkeletonBlocks (загрузка), EmptyStateCard (пустые состояния), спиннеры на формах.

## Глава 3. Разработка, реализация и тестирование веб-приложения

**3.1. Реализация серверной части**
- Структура backend-проекта: main.py, config.py, database.py, models/, schemas/, api/, services/.
- Точка входа FastAPI: CORS, статика, подключение роутеров.
- ORM-модели: определение таблиц, связи, каскадное удаление.
- Сервис проверки конфликтов (`booking_conflict.py`): алгоритм, буфер 15 минут.
- Бизнес-правила отмены: дедлайн 30 минут, обработка timezone.

**3.2. Аутентификация, авторизация и Telegram-интеграция**
- Процесс регистрации: нормализация Telegram-поля (ник / телефон), bcrypt, создание пользователя.
- JWT flow: выдача токена при входе, хранение в localStorage, заголовок Authorization.
- Зависимости FastAPI: `get_current_user`, `require_admin`.
- Telegram Webhook (`/api/telegram/webhook`):
  - Обработка `/start <payload>` — поиск пользователя, генерация и отправка кода.
  - Обработка `contact`-сообщения — верификация по номеру телефона.

**3.3. Реализация клиентской части**
- Структура frontend-проекта: api/, components/, context/, hooks/, pages/.
- HTTP-клиент: `apiFetch`, `buildApiUrl`, `mediaUrl` — корректное разрешение URL в dev и production.
- AuthContext: хранение токена, автозагрузка профиля, login/logout.
- Ключевые страницы: дашборд с избранным и историей, страница комнаты с FullCalendar, детали брони.
- Административная панель: CRUD комнат, загрузка фото с превью, управление ролями.

**3.4. Тестирование системы**
- Стратегия: модульное + интеграционное тестирование.
- Backend (Pytest): тест-кейсы для конфликтов бронирований (буфер, граничные случаи), политики отмены, контроля доступа (403 для постороннего).
- Frontend (Vitest + happy-dom): тесты API-клиента (Authorization-заголовок, ApiError), рендеринг HomePage.
- Таблицы тест-кейсов с ожидаемыми результатами.
- Итоги тестирования: все тесты проходят.

**3.5. Развёртывание приложения**
- Выбор платформы: Render (бесплатный план, managed PostgreSQL, Blueprint).
- Конфигурация `render.yaml`: backend (Web Service), frontend (Static Site), database.
- Переменные окружения: SECRET_KEY, TELEGRAM_BOT_TOKEN, DATABASE_URL, VITE_API_URL, CORS_ORIGINS.
- Этапы первого деплоя: подключение GitHub → Blueprint → ENV → миграции → регистрация Telegram webhook.
- Схема взаимодействия сервисов на Render.
