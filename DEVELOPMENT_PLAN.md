# План разработки: Веб-приложение бронирования переговорных комнат

## Оценка проекта и сроков (30 дней)

**Вердикт: реалистично при 4–6 часах работы в день и чётком соблюдении плана.**

| Фактор | Оценка |
|--------|--------|
| Объём функционала | Средний, типичный учебный CRUD + календарь |
| Стек (FastAPI + React) | Хорошо документирован, много примеров |
| Риски | Календарь и проверка конфликтов — самые трудоёмкие части |
| Резерв | ~5–7 дней на доработки и защиту |

Рекомендация: первые 2 недели — бекенд + API + тесты; следующие 2 недели — фронтенд + интеграция + полировка.

## Этап 2: Авторизация (Дни 4–5)

### Backend
- [ ] Регистрация: POST `/api/auth/register` (email, password, full_name) → создание User, хэш пароля (bcrypt)
- [ ] Вход: POST `/api/auth/login` (email, password) → JWT access_token (и при необходимости refresh)
- [ ] Зависимость `get_current_user`: проверка JWT, загрузка User
- [ ] Роль админа: зависимость `require_admin` (проверка `user.is_admin`)
- [ ] Swagger: описание эндпоинтов (FastAPI даёт OpenAPI из коробки)

### Frontend
- [ ] Страницы: Login, Register
- [ ] AuthContext: хранение токена и пользователя, логин/логаут
- [ ] Защита маршрутов: редирект на логин для приватных страниц
- [ ] Запросы к API с заголовком `Authorization: Bearer <token>`

**Результат этапа:** пользователь может зарегистрироваться, войти и выйти; админ определяется по флагу.

## Этап 4: Бронирования и проверка конфликтов (Дни 9–14) — ключевой

### Backend (самое важное)
- [ ] **Проверка пересечения времени:**
  - Два слота пересекаются, если: `start_time < existing.end_time AND end_time > existing.start_time`
  - Сервис/функция: по `room_id`, `start_time`, `end_time` (и опционально `exclude_booking_id`) проверить наличие конфликтующих записей в БД
- [ ] Создание брони: POST `/api/bookings` (room_id, start_time, end_time)
  - Проверка авторизации
  - Валидация: end_time > start_time, время в будущем (или по правилам)
  - Вызов проверки конфликтов → при конфликте 409 Conflict
  - Сохранение брони, привязка к текущему пользователю
- [ ] Список бронирований комнаты: GET `/api/rooms/{id}/bookings` или GET `/api/bookings?room_id=...&from=&to=` для календаря
- [ ] Мои бронирования: GET `/api/bookings/me` (только будущие или с фильтром)
- [ ] Отмена: DELETE `/api/bookings/{id}` с проверкой, что бронь принадлежит пользователю и её ещё можно отменить (например только будущие)

### Unit-тесты (Pytest)
- [ ] Тесты на логику пересечения интервалов (несколько кейсов: нет пересечения, полное включение, частичное)
- [ ] Тесты API: создание брони при конфликте → 409; без конфликта → 201
- [ ] Тест отмены чужой брони → 403

### Frontend
- [ ] Календарь на странице комнаты: FullCalendar (или аналог), показ занятых слотов из GET `/api/rooms/{id}/bookings` (или по диапазону)
- [ ] Форма создания брони: выбор даты и времени (или слоты), отправка POST `/api/bookings`, обработка 409 с сообщением «Время занято»
- [ ] Страница «Мои бронирования»: список будущих броней, кнопка «Отменить»

**Результат этапа:** создание брони с проверкой конфликтов, календарь комнаты, «Мои бронирования» и отмена. Есть unit-тесты на конфликты.

## Риски и советы

1. **Конфликты** — вынести логику пересечения в отдельную функцию/сервис и покрыть тестами в первую очередь.
2. **Календарь** — выбрать одну библиотеку (FullCalendar) и минимальный набор (месяц/неделя, события из API).
3. **Сроки** — если отстаёте, упростить админку (только добавление/удаление комнат без сложного редактирования) и декорации UI.
4. **JWT** — хранить в памяти или httpOnly cookie, не в localStorage при желании усилить безопасность (для диплома достаточно и localStorage).

Удачи с дипломом.
