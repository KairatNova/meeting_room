## 2. План по шагам (бэкенд → фронтенд)

Работать в таком порядке: сначала доводим API до рабочего состояния, затем подключаем и дорабатываем фронт.

---

### Шаг 1. Backend: комнаты (CRUD + список с фильтрами)

**Цель:** список комнат и детальная комната работают; админ может создавать/редактировать/удалять комнаты.

- **Rooms:**
  - `GET /api/rooms` — выборка из БД с фильтрами `capacity_min`, `search` (по `name`/`description`), вернуть `list[RoomResponse]`.
  - `GET /api/rooms/{room_id}` — одна комната по id, 404 если нет.
  - `POST /api/rooms` — создание (AdminUser), вернуть 201 + RoomResponse.
  - `PATCH /api/rooms/{room_id}` — частичное обновление (AdminUser).
  - `DELETE /api/rooms/{room_id}` — удаление (AdminUser), 204.

**Результат:** Swagger проверяется вручную; фронт «Список комнат» и «Детальная комната» начинают показывать данные.

---

### Шаг 2. Backend: проверка конфликтов и бронирования

**Цель:** одна точка истины для «пересечения слотов» + создание/отмена брони.

- **Сервис конфликтов** (например `app/services/booking_conflict.py` или в `app/services/booking.py`):
  - Функция: по `room_id`, `start_time`, `end_time` и опционально `exclude_booking_id` проверить, есть ли в БД брони с пересечением.
  - Пересечение: `start_time < existing.end_time AND end_time > existing.start_time`.
  - Покрыть unit-тестами (pytest): нет пересечения, полное включение, частичное пересечение, граница в одну минуту.

- **Bookings API:**
  - `GET /api/bookings` — список бронирований с фильтрами `room_id`, `from_time`, `to_time` (для календаря по комнате).
  - `POST /api/bookings` — создание (CurrentUser): валидация `end_time > start_time`, опционально «в будущем»; вызов проверки конфликтов → при конфликте **409 Conflict**; сохранить, вернуть 201 + BookingResponse.
  - `GET /api/bookings/me` — брони текущего пользователя (фильтр по `from_time`/`to_time` при необходимости).
  - `DELETE /api/bookings/{booking_id}` — отмена: только владелец (`user_id == current_user.id`), только будущие (`end_time > now()`); 204 или 403/404.

**Результат:** создание брони с конфликтом даёт 409; без конфликта — 201. Мои брони и отмена работают. В Swagger всё видно.

---

### Шаг 3. Backend: улучшения API (по желанию)

- Валидация: например, не допускать брони в прошлое; ограничение максимальной длительности слота.
- В `BookingResponse` при необходимости добавить вложенные `room`/`user` (или отдельные эндпоинты), чтобы фронт не дергал лишние запросы.
- Описание в OpenAPI: теги, примеры, описание 409 для конфликтов.

---

### Шаг 4. Frontend: список комнат и детальная страница

**Цель:** после шага 1 бэкенда — реальные данные на экране.

- Убедиться, что запросы идут на `/api/rooms` и `/api/rooms/:id` (уже есть в `roomsApi`).
- Список: уже есть фильтры и карточки; при успешном ответе бэкенда всё заработает.
- Детальная страница: показ комнаты уже есть; добавить секцию «Календарь» (см. шаг 5) и блок «Создать бронирование» (форма дата/время или слоты).

---

### Шаг 5. Frontend: календарь и создание брони

**Цель:** календарь доступности одной комнаты + создание брони с обработкой 409.

- Установить **FullCalendar** (или аналог) и отображать события из `GET /api/bookings?room_id=<id>&from_time=&to_time=` в формате, подходящем для календаря (например, маппинг в `{ id, title, start, end }`).
- Форма брони: выбор даты и времени (или слотов); отправка `POST /api/bookings` с `room_id`, `start_time`, `end_time`.
- При **409** показывать сообщение «Время занято» / «Выберите другой слот».
- После успешного создания — обновить календарь и/или список «Мои бронирования».

---

### Шаг 6. Frontend: «Мои бронирования» и отмена

**Цель:** личный кабинет с отменой.

- Страница уже дергает `GET /api/bookings/me` и `DELETE /api/bookings/:id`; после реализации бэкенда (шаг 2) достаточно доработать UX:
  - Показывать название комнаты (если бэкенд вернёт вложенный объект или отдельно подтянуть комнаты).
  - Показывать только будущие или с фильтром по датам.
  - При отмене — подтверждение и обновление списка; обработка ошибок (403/404).

---

### Шаг 7. Frontend: админ-панель (комнаты)

**Цель:** админ может добавлять/редактировать/удалять комнаты.

- Форма создания комнаты: название, описание, вместимость, удобства; `roomsApi.create`.
- Список комнат в админке: таблица или карточки; кнопки «Изменить» / «Удалить».
- Форма редактирования: предзаполнить из `roomsApi.get(id)`, отправить `roomsApi.update(id, data)`.
- Удаление: подтверждение, затем `roomsApi.delete(id)`.

---

### Шаг 8. Адаптивность и полировка

- Проверить все страницы на мобильных брейкпоинтах (Tailwind уже подключён).
- Обработка ошибок: единообразные сообщения с бэкенда (401, 403, 404, 409, 422).
- Загрузки: спиннеры/скелетоны при запросах.
- README: как запустить backend (uvicorn), frontend (npm run dev), БД (PostgreSQL + миграции), `.env.example`.

## 4. Краткий порядок работ

1. **Backend: комнаты** — реализовать все эндпоинты в `app/api/rooms.py` (запросы к БД, 404 для отсутствующей комнаты).
2. **Backend: сервис конфликтов** — функция проверки пересечения + тесты.
3. **Backend: бронирования** — реализовать list/create/me/cancel в `app/api/bookings.py` с вызовом проверки конфликтов и 409.
4. **Frontend: комнаты** — убедиться, что список и детальная страница работают с новым API.
5. **Frontend: календарь + форма брони** — FullCalendar, запросы к `/api/bookings`, обработка 409.
6. **Frontend: мои бронирования** — доработать отображение и отмену.
7. **Frontend: админ** — CRUD комнат (формы и вызовы API).
8. **Полировка** — адаптив, ошибки, README.

После шагов 1–3 бэкенд полностью закрывает требования по API; шаги 4–8 приводят фронт и UX в соответствие с дипломным заданием.
