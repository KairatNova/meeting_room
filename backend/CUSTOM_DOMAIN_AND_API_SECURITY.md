# Короткий домен и защита API

## 1. Короткий URL вместо длинного Railway

Сейчас у вас что-то вроде:
- Фронт: `https://brave-courtesy-production.up.railway.app`
- Бэкенд: `https://meetingroom-production-15c9.up.railway.app`

**Вариант A: Свой домен в Railway (рекомендуется)**

1. Купите домен (например, на Reg.ru, Namecheap, Cloudflare) или возьмите бесплатный поддомен (например, через Freenom или ваш проваейдер).
2. В **Railway** откройте проект → сервис (фронт или бэкенд) → вкладка **Settings** → **Networking** / **Domains**.
3. Нажмите **Add Custom Domain** и введите, например:
   - для фронта: `app.ваш-домен.ru` или `meeting.ваш-домен.ru`;
   - для бэкенда: `api.ваш-домен.ru`.
4. Railway покажет, что прописать в DNS (обычно CNAME на `xxx.up.railway.app` или A-запись на их IP). Пропишите это у регистратора домена.
5. После распространения DNS (от минут до суток) ваш сервис будет доступен по короткому адресу.

Итог: вместо длинного URL будет, например, `https://app.ваш-домен.ru` и `https://api.ваш-домен.ru`.

**Вариант B: Без своего домена**

Короткий «красивый» URL без своего домена Railway не даёт. Можно использовать бесплатный сервис вроде **Vercel** или **Netlify** для фронта — там часто выдают короткие поддомены вида `ваш-проект.vercel.app`.

## 2. Защита бэкенда: чтобы не всем был доступен API

Цель: чтобы с `https://meetingroom-production-15c9.up.railway.app` не мог воспользоваться любой человек с улицы (скрипты, Postman и т.п.).

### Что уже есть: CORS

В `.env` бэкенда задаётся **CORS_ORIGINS** — список разрешённых источников запросов. Укажите **только ваш фронт**:

```env
CORS_ORIGINS=https://brave-courtesy-production.up.railway.app
```

Или после настройки своего домена:

```env
CORS_ORIGINS=https://app.ваш-домен.ru
```

Тогда запросы из браузера с других сайтов браузер заблокирует. Но **CORS не мешает** вызовам с Postman, curl или со своего сервера — они не отправляют Origin или могут подделать заголовок.

### Дополнительно: опциональный API-ключ

В коде добавлена **опциональная** проверка заголовка **X-API-Key**: если в `.env` задан **ALLOWED_API_KEY**, то все запросы к API (кроме `/health`, webhook Telegram и документации) должны содержать этот ключ в заголовке, иначе бэкенд вернёт 403.

- Защищает от случайного использования вашего API по прямой ссылке (знать URL мало — нужен ещё ключ).
- Ключ будет в коде фронта (переменная окружения), поэтому от целевого разбора фронта это не спасает, но от простых скриптов и ботов — да.

**Как включить:**

1. В **backend** `.env` добавьте (придумайте длинный случайный ключ):

```env
ALLOWED_API_KEY=ваш-длинный-секретный-ключ-минимум-20-символов
```

2. Во **frontend** в переменных окружения (Railway / Vite) задайте тот же ключ, например:

```env
VITE_API_KEY=ваш-длинный-секретный-ключ-минимум-20-символов
```

3. Во фронте все запросы к API должны отправлять заголовок:

```http
X-API-Key: <значение VITE_API_KEY>
```

(В проекте это можно сделать в общем клиенте запросов, например в `api/client.ts` или аналоге.)

**Исключения** (проверка ключа не применяется):

- `GET /health` — для мониторинга и проверки работы сервиса;
- `POST /api/telegram/webhook` — Telegram не пришлёт ваш ключ;
- `/docs`, `/redoc`, `/openapi.json` — при необходимости их можно отключить в проде через настройки приложения.

### Рекомендации

- Обязательно: задайте **CORS_ORIGINS** только на ваш фронт (и при смене домена обновите).
- По желанию: включите **ALLOWED_API_KEY** и добавьте **X-API-Key** во все запросы фронта к бэкенду.
- Для продакшена: отключите или ограничьте доступ к `/docs` и `/redoc`, если не хотите светить API наружу.

После этого «просто так» воспользоваться API по адресу `https://meetingroom-production-15c9.up.railway.app` будет сложнее: без ключа запросы к основным эндпоинтам будут отклоняться.
