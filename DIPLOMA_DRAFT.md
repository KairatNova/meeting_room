> This thesis is devoted to the development of a web application for automating the meeting room booking process in organizations.
>
> The work includes an analysis of the subject domain and existing analogues, formulation of functional and non-functional requirements, architecture and database design, as well as implementation of both server-side and client-side parts of the application.
>
> The backend is built using the FastAPI framework with PostgreSQL as the database management system and SQLAlchemy as the ORM layer. ### 1.1. Описание предметной области

📌 **Что писать:** Опишите процесс бронирования в организации. Кто участвует, какие данные нужны, какие проблемы существуют без автоматизации.  

✏️ **Черновик:**

> Переговорная комната (meeting room) — выделенное помещение в офисе или учебном заведении, предназначенное для проведения встреч, переговоров, презентаций или рабочих сессий. В организациях с несколькими комнатами возникает необходимость управления их расписанием.
>
> Типичный процесс бронирования без автоматизации выглядит следующим образом:
> 1. Сотрудник узнаёт устно или по таблице, свободна ли комната.
> 2. Записывает своё имя и время в общий журнал или таблицу.
> 3. При нарушении порядка записей возможны конфликты.
>
> Основные проблемы ручного подхода:
> - **Двойные бронирования** — два человека одновременно занимают одну комнату.
> - **Отсутствие прозрачности** — нет единой точки отображения расписания.
> - **Нет контроля отмен** — пользователи забывают отменить бронь, комната пустует.
> - **Нет истории** — сложно анализировать загруженность.
>
> Ключевые сущности предметной области:
> - **Пользователь** (User) — сотрудник или студент, который бронирует комнату.
> - **Переговорная комната** (Room) — ресурс с параметрами: название, вместимость, местоположение, оборудование, фотографии.
> - **Бронирование** (Booking) — факт занятости комнаты: кто, когда и на какое время.
> - **Администратор** — пользователь с расширенными правами: управление комнатами и ролями.

**Серверная часть:**

> Для реализации REST API был выбран **FastAPI** (Python). FastAPI обеспечивает автоматическую валидацию запросов через Pydantic-схемы, автогенерацию Swagger-документации и высокую производительность за счёт асинхронной обработки запросов (ASGI). Альтернативами рассматривались Django REST Framework (избыточный для MVP) и Express.js (Node.js, требует отдельного изучения).
>
> В качестве СУБД выбрана **PostgreSQL** — зрелая реляционная база данных с поддержкой транзакций, индексов и сложных запросов. Для работы с базой применяется **SQLAlchemy** как ORM, обеспечивающий абстракцию над SQL и удобное определение моделей на Python. Версионирование схемы БД реализовано через **Alembic**.

**Клиентская часть:**

> Фронтенд реализован на **React** с **TypeScript**. React обеспечивает компонентную архитектуру и богатую экосистему, TypeScript — типобезопасность и снижение числа ошибок при разработке. В качестве сборщика выбран **Vite** (быстрее Create React App, нативная поддержка ESM). Стилизация выполнена через **Tailwind CSS** — утилитарный CSS-фреймворк, ускоряющий адаптивную вёрстку.

**Сторонние интеграции:**

> Для отображения календаря занятости использована библиотека **FullCalendar** — профессиональное решение для визуализации событий по дням и неделям. Для верификации аккаунтов интегрирован **Telegram Bot API**, позволяющий доставлять коды подтверждения через мессенджер, популярный среди целевой аудитории.

> 1. **Клиентская часть (Frontend SPA)** — одностраничное приложение (Single Page Application) на React, запускаемое в браузере пользователя. Взаимодействует с сервером через REST API по протоколу HTTPS.
>
> 2. **Серверная часть (Backend API)** — Python-приложение на FastAPI, обрабатывает HTTP-запросы, реализует бизнес-логику, обращается к базе данных. Предоставляет REST API и Swagger-документацию.
>
> 3. **База данных (PostgreSQL)** — хранит данные о пользователях, комнатах, бронированиях и кодах верификации.
>
> Дополнительно взаимодействует с **Telegram Bot API** (внешний сервис) для отправки кодов верификации пользователям.

```
[Браузер / React SPA]
        |
        | HTTPS REST API
        ↓
[FastAPI Backend]  ←→  [PostgreSQL DB]
        |
        | Telegram Bot API (webhooks / HTTP)
        ↓
[Telegram Servers] → [Telegram App пользователя]
```

### 3.2. Аутентификация и авторизация

📌 **Что писать:** Описать процесс регистрации, выдачи JWT, защиту маршрутов.  

✏️ **Черновик:**

> **Процесс регистрации:**
> 1. Клиент отправляет POST `/api/auth/register` с полями `name`, `email`, `password`, `telegram` (ник `@username` или номер телефона).
> 2. Backend валидирует данные через Pydantic-схему `UserCreate`, нормализует Telegram-поле.
> 3. Пароль хэшируется через `bcrypt`.
> 4. Создаётся запись `User` в БД с `is_verified=False`.
> 5. Backend ищет Telegram-бота (если передан ник — по нику, если номер — ждёт `contact`-сообщение), отправляет 6-значный код.
> 6. Пользователь вводит код в интерфейсе, POST `/api/auth/verify-telegram` — `is_verified=True`.
>
> **JWT-авторизация:**
> - При входе (POST `/api/auth/login`) проверяется email/логин + код. При успехе возвращается `access_token` (JWT).
> - Клиент хранит токен в `localStorage` и передаёт в заголовке `Authorization: Bearer <token>`.
> - Зависимость `get_current_user` (FastAPI Depends) декодирует токен при каждом запросе к защищённым маршрутам.
> - Зависимость `require_admin` дополнительно проверяет `is_admin=True`.
>
> **Пример защищённого маршрута:**

```python
@router.get("/me")
async def get_me(current_user: User = Depends(get_current_user)):
    return current_user
```

### 3.3. Управление комнатами и бронированиями

📌 **Что писать:** Бизнес-логика: проверка конфликтов, буфер, правила отмены. Привести фрагменты кода.  

✏️ **Черновик:**

> **Проверка конфликтов бронирований:**
>
> Ключевая бизнес-функция — `has_booking_conflict` в `services/booking_conflict.py`. Функция проверяет, пересекается ли запрашиваемый интервал с уже существующими активными бронированиями для данной комнаты. Дополнительно применяется буфер 15 минут: интервал новой брони расширяется на 15 минут с каждой стороны перед проверкой пересечений.
>
> При обнаружении конфликта API возвращает ответ `409 Conflict`.
>
> **Правила отмены:**
>
> Отмена бронирования (DELETE `/api/bookings/{id}`) запрещена, если до начала брони остаётся менее 30 минут. При нарушении этого правила API возвращает `400 Bad Request` с описанием причины.

```python
# Фрагмент из bookings.py
CANCEL_DEADLINE_MINUTES = 30
now = datetime.now(timezone.utc)
if _ensure_utc(booking.start_time) - now < timedelta(minutes=CANCEL_DEADLINE_MINUTES):
    raise HTTPException(status_code=400,
        detail="Отмена доступна не позднее чем за 30 минут до начала")
```

### 3.4. Интеграция с Telegram Bot API

📌 **Что писать:** Как работает вебхук, как бот отправляет коды, как обрабатывает contact-сообщения.  

✏️ **Черновик:**

> Telegram-бот выступает средством верификации аккаунтов. Схема работы:
>
> 1. При регистрации пользователь указывает Telegram-ник или номер телефона.
> 2. Ссылка на бот (`t.me/<BOT_USERNAME>?start=<TOKEN>`) отображается на странице верификации.
> 3. Пользователь открывает бота и нажимает /start — бот находит его по нику и отправляет 6-значный код.
> 4. Если указан номер телефона, бот запрашивает отправку контакта (кнопка "Поделиться номером"). После получения контакта бот сопоставляет номер с зарегистрированным пользователем и отправляет код.
> 5. Пользователь вводит код на сайте — аккаунт подтверждается.
>
> Обработка входящих сообщений реализована через вебхук: Telegram отправляет POST-запросы на `/api/telegram/webhook`. После деплоя вебхук регистрируется через Telegram Bot API: `setWebhook?url=<BACKEND_URL>/api/telegram/webhook`.
