### 1.1. Описание предметной области и проблематика

📌 **Что писать:** Что такое переговорная комната, кто участники процесса, как выглядит процесс бронирования, какие проблемы без автоматизации.  

✏️ **Черновик:**

> Переговорная комната (англ. *meeting room*) — специально оборудованное помещение для проведения деловых встреч, переговоров, совещаний, обучающих мероприятий и презентаций. В организациях со значительным числом сотрудников таких комнат может быть несколько, каждая с различной вместимостью, техническим оснащением и местоположением.
>
> **Участники процесса:**
> - **Пользователь (сотрудник/студент)** — инициирует бронирование, выбирает комнату и временной слот.
> - **Администратор** — управляет перечнем комнат, их параметрами и правами доступа.
>
> **Типичный процесс бронирования без автоматизации:**
> 1. Сотрудник узнаёт о свободной комнате устно или через общий файл.
> 2. Вносит запись в журнал (бумажный или Excel-таблицу).
> 3. При необходимости уведомляет коллег о занятости.
> 4. При отмене — самостоятельно удаляет запись.
>
> **Проблемы ручного подхода:**
>
> | Проблема | Последствие |
> |----------|-------------|
> | Двойные бронирования | Конфликты между сотрудниками, срыв встреч |
> | Нет единого расписания | Сложно найти свободное время |
> | Нет контроля отмен | Комнаты пустуют, хотя числятся занятыми |
> | Нет истории | Невозможно анализировать загруженность |
> | Нет уведомлений | Пользователи забывают о встречах |
>
> **Ключевые сущности предметной области:**
> - **Пользователь (User)** — имя, email, Telegram-контакт, роль (пользователь / администратор).
> - **Переговорная комната (Room)** — название, вместимость, местоположение, оборудование, фотографии, статус активности.
> - **Бронирование (Booking)** — ссылки на комнату и пользователя, дата/время начала и конца, статус.
> - **Код верификации (VerificationCode)** — 6-значный код с ограниченным сроком действия для верификации аккаунта.

### 1.2. Обзор и сравнительный анализ существующих аналогов

📌 **Что писать:** 4–5 аналогов с кратким описанием, сравнительная таблица, вывод.  

✏️ **Черновик:**

> Для обоснования разработки собственного решения был проведён анализ пяти существующих систем бронирования.
>
> **Google Calendar + Room Resources** — инструмент, встроенный в экосистему Google Workspace. Позволяет создавать ресурсы (комнаты) и управлять бронированиями через стандартный интерфейс календаря. Требует корпоративной подписки Google Workspace, не имеет интеграции с Telegram, интерфейс не специализирован под бронирование комнат.
>
> **Microsoft Outlook Room Booking** — функция корпоративного пакета Microsoft 365. Поддерживает ресурсные ящики, делегирование, сложные политики. Требует лицензии M365, не имеет Telegram-интеграции, ориентирован на крупные корпорации.
>
> **Robin Powered** — специализированная SaaS-платформа для управления офисом, включая переговорные комнаты. Имеет красивый интерфейс, аналитику, мобильное приложение. Полностью платный, без русского языка, без Telegram.
>
> **Skedda** — облачный инструмент для бронирования помещений. Есть бесплатный план с ограничениями. Отсутствует русский язык и Telegram-интеграция. Слабая настраиваемость бизнес-правил.
>
> **Колокол / собственные Excel-таблицы** — типичное решение в небольших организациях. Бесплатно, но лишено всех преимуществ автоматизации.

### 1.4. Выбор и обоснование технологического стека

📌 **Что писать:** Почему выбраны именно эти технологии, сравнение с альтернативами.  

✏️ **Черновик:**

> При выборе технологий оценивались следующие критерии: зрелость и поддержка сообществом, скорость разработки, бесплатная доступность, производительность, соответствие задачам проекта.

**Серверная часть (Backend):**

> Для реализации REST API выбран **FastAPI** (Python 3.11). Среди ключевых преимуществ: автоматическая валидация запросов и ответов через Pydantic, автогенерация Swagger/OpenAPI-документации, нативная поддержка асинхронных операций (ASGI), высокая производительность. Альтернативами рассматривались Django REST Framework (избыточен для проекта данного масштаба, медленнее в разработке API) и Express.js (Node.js — требует освоения отдельного языка).
>
> В качестве СУБД выбрана **PostgreSQL 16** — зрелая реляционная база данных с поддержкой транзакций ACID, мощными индексами и широкой поддержкой облачными платформами. Для работы с БД применяется **SQLAlchemy 2.0** (ORM), обеспечивающий удобное определение моделей на Python и гибкость при написании запросов. Управление версиями схемы — **Alembic**.

**Клиентская часть (Frontend):**

> Фронтенд реализован на **React 18** с **TypeScript**. React обеспечивает компонентную архитектуру и богатую экосистему библиотек; TypeScript добавляет статическую типизацию, снижая число ошибок в runtime. Сборка — **Vite** (значительно быстрее устаревшего CRA за счёт ESM и esbuild). Стилизация — **Tailwind CSS**: утилитарный CSS-фреймворк ускоряет адаптивную вёрстку без разрастания кастомных стилей.
>
> Для отображения расписания интегрирована библиотека **FullCalendar** — профессиональное решение с поддержкой видов "неделя" и "день", drag-and-drop (опционально) и гибкой настройкой событий.

**Аутентификация и интеграции:**

### 3.2. Аутентификация, авторизация и Telegram-интеграция

📌 **Что писать:** JWT flow, процесс верификации, Telegram webhook.  

✏️ **Черновик:**

**Регистрация и верификация:**

> Процесс регистрации включает несколько шагов:
> 1. Клиент отправляет `POST /api/auth/register` с полями `name`, `email`, `password`, `telegram` (ник `@username` или номер `+7...`).
> 2. Backend через `model_validator` в схеме `UserCreate` нормализует `telegram`-поле: если начинается с `@` или буквы — сохраняется как `telegram_username`, если содержит только цифры и `+` — как `phone`.
> 3. Пароль хэшируется через `bcrypt` (`passlib`). Создаётся запись `User` с `is_verified=False`.
> 4. Клиент получает ссылку `t.me/<BOT_USERNAME>?start=<email_base64>` и инструкцию открыть бота.
> 5. Пользователь открывает бота, нажимает Start — бот получает webhook с `text=/start <payload>`, декодирует email, ищет пользователя, отправляет 6-значный код.
> 6. Если пользователь регистрировался с номером телефона, бот предлагает нажать кнопку «Поделиться номером». После получения `contact`-сообщения бот сверяет номер с зарегистрированным и отправляет код.
> 7. Пользователь вводит код на странице `/verify-telegram` — `POST /api/auth/verify-telegram` → `is_verified=True`.

**JWT-авторизация:**

> После верификации пользователь выполняет вход: `POST /api/auth/login`. Backend проверяет логин (email / ник / телефон) и присылает 6-значный код через бота. При подтверждении кода выдаётся `access_token` (JWT, подписан `SECRET_KEY`, алгоритм `HS256`, срок действия задаётся в настройках).
>
> Клиент сохраняет токен в `localStorage` и передаёт в каждом запросе: `Authorization: Bearer <token>`.
>
> Зависимость FastAPI `get_current_user` декодирует токен, извлекает `user_id` и возвращает объект пользователя. `require_admin` дополнительно проверяет `user.is_admin == True`.

**Telegram Webhook:**

```
POST /api/telegram/webhook
← вызывается Telegram-серверами при каждом новом сообщении боту
```

> Обработчик разбирает тип сообщения:
> - `/start <payload>` — декодирует email из payload, находит пользователя, генерирует и отправляет код.
> - `contact` (сообщение с телефоном) — нормализует номер, ищет пользователя по `phone`, отправляет код.
> - Прочие сообщения — ответ-подсказка.

**HTTP-клиент и URL-разрешение:**

> Базовый клиент (`api/client.ts`) реализует функцию `apiFetch`, которая автоматически добавляет заголовок `Authorization` из `localStorage`, обрабатывает JSON-ответы и возбуждает `ApiError` при HTTP-ошибках.
>
> Функция `buildApiUrl(path)` формирует абсолютный URL к API с учётом переменной окружения `VITE_API_URL`. В режиме разработки используется прокси Vite (`/api → http://localhost:8000`), в production — реальный URL backend-сервиса на Render. Функция `mediaUrl(src)` аналогично разрешает URL статических файлов (фотографий комнат).

**Аутентификационный контекст:**

> `AuthContext` предоставляет через React Context: объект `user` (данные текущего пользователя или `null`), флаг `isLoading`, функции `login`, `register`, `logout`. При загрузке страницы контекст проверяет наличие токена в `localStorage` и автоматически загружает профиль (`GET /api/auth/me`). Это обеспечивает сохранение сессии при перезагрузке страницы.

**Защита маршрутов:**

> - `PrivateRoute` — перенаправляет на `/login`, если пользователь не авторизован.
> - `AdminRoute` — перенаправляет на `/`, если пользователь не имеет роли `is_admin`.

**Интеграция FullCalendar:**

> На странице комнаты (`RoomDetailPage.tsx`) компонент `FullCalendar` конфигурируется с видами `timeGridWeek` и `timeGridDay`. При монтировании загружаются бронирования через `GET /api/bookings?room_id=<id>` и отображаются как события. Форма бронирования обновляет календарь после успешного создания брони.

### 3.5. Развёртывание приложения

📌 **Что писать:** Платформа, Blueprint, этапы деплоя, ENV-переменные.  

✏️ **Черновик:**

**Выбор платформы:**

> Для развёртывания выбрана платформа **Render** (render.com). Платформа предоставляет: бесплатный план для Web Service и Static Site, managed PostgreSQL, автодеплой из GitHub, поддержку декларативной конфигурации через Blueprint (`render.yaml`).

**Конфигурация Blueprint (`render.yaml`):**

> Файл `render.yaml` в корне репозитория описывает все сервисы декларативно, что позволяет развернуть всю инфраструктуру одним кликом через интерфейс Render:
>
> - **`meeting-room-db`** — Managed PostgreSQL, план Free.
> - **`meeting-room-backend`** — Web Service, Python, команда запуска `uvicorn app.main:app --host 0.0.0.0 --port $PORT`, переменные окружения (включая автоматическую привязку к БД).
> - **`meeting-room-frontend`** — Static Site, команда сборки `npm run build`, папка `dist`, переменная `VITE_API_URL` задаётся вручную.

**Процесс первого развёртывания:**

```
1. Подключить репозиторий GitHub к аккаунту Render.
2. Render обнаруживает render.yaml → предлагает создать Blueprint.
3. После создания сервисов задать секретные ENV-переменные:
   backend: SECRET_KEY, TELEGRAM_BOT_TOKEN, TELEGRAM_BOT_USERNAME
   frontend: VITE_API_URL = https://<backend-service>.onrender.com
4. Зарегистрировать Telegram webhook:
   GET https://api.telegram.org/bot<TOKEN>/setWebhook
       ?url=https://<backend>.onrender.com/api/telegram/webhook
5. Выполнить миграции БД:
   render run --service <backend-service-id> -- alembic upgrade head
6. Создать первого администратора через Render Shell:
   UPDATE users SET is_admin=true WHERE email='admin@example.com';
```

**Схема взаимодействия сервисов на Render:**

```
GitHub (main branch)
       │  autopush
       ▼
┌──────────────────┐    ┌─────────────────────┐
│ Frontend (Static)│    │ Backend (Web Service)│
│ Vite build → CDN │    │ FastAPI + uvicorn     │
└──────────────────┘    └──────────┬──────────┘
  VITE_API_URL ──────────────────► │
                                   │ DATABASE_URL
                         ┌─────────▼──────────┐
                         │  PostgreSQL (Render)│
                         └────────────────────┘
```
