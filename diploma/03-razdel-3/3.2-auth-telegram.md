3.2. Аутентификация, авторизация и интеграция с Telegram

Подсистема учёта доступа реализует регистрацию с подтверждением личности через мессенджер, выдачу JWT и восстановление пароля. Логика сосредоточена в роутере api/auth.py, обработчике webhook и модуле services/telegram.py.

Регистрация и подтверждение аккаунта

При POST /api/auth/register клиент передаёт имя, email, пароль и идентификатор Telegram (ник с @ или номер телефона). Сервер нормализует email, разбирает поле Telegram на username или phone, хэширует пароль и создаёт пользователя с неподтверждённым статусом.

Генерируется шестизначный цифровой код и запись в таблице email_verification_codes с ограниченным сроком действия. Отдельно создаётся одноразовый token в telegram_pending_links; клиенту возвращается deep-link вида t.me/ИмяБота?start=TOKEN. На электронную почту коды не отправляются — канал доставки только Telegram.

Пользователь открывает бота и нажимает Start. Webhook получает команду /start TOKEN, находит запись по токену, сохраняет telegram_chat_id у пользователя и отправляет код через Bot API. Если при регистрации указан телефон без ника, бот дополнительно запрашивает контакт; после сверки номера код отправляется повторно.

На странице /verify-email клиент вводит email и код; POST /api/auth/verify-email при успехе помечает учётную запись как подтверждённую, удаляет использованный код и сразу возвращает JWT.

Вход в систему

Два сценария.

Простой вход: POST /api/auth/login по email и паролю. Проверяется хэш bcrypt и факт подтверждения аккаунта; в ответ — access_token (JWT, алгоритм HS256, срок из настроек).

Вход с дополнительным кодом: POST /api/auth/login-request принимает логин (email, ник или телефон) и пароль. При успехе в Telegram уходит код из таблицы login_verification_codes; POST /api/auth/login-verify проверяет код и выдаёт JWT. Отправка возможна только если chat_id уже привязан при регистрации.

Клиент сохраняет токен в localStorage и передаёт в заголовке Authorization: Bearer. Зависимость get_current_user декодирует JWT, извлекает идентификатор пользователя из поля sub и загружает запись из БД. Для админ-маршрутов вызывается require_admin.

Восстановление пароля

POST /api/auth/forgot-password по email или Telegram-нику создаёт или обновляет запись в password_reset_codes и отправляет код в Telegram, если chat_id известен. Ответ унифицирован, чтобы не раскрывать наличие аккаунта.

POST /api/auth/reset-password принимает логин, код и новый пароль; при совпадении кода пароль перезаписывается, код удаляется.

Интеграция с Telegram Bot API

Модуль services/telegram.py выполняет HTTPS-запросы к api.telegram.org методом sendMessage (библиотека httpx). Формируются тексты для кода регистрации и сброса пароля. Функция generate_link_token создаёт URL-safe токен длиной около 43 символов для параметра start.

Эндпоинт POST /api/telegram/webhook принимает JSON Update от серверов Telegram. Обработчик разбирает /start с токеном и сообщения contact, ведёт журнал событий, всегда отвечает {"ok": true}, чтобы Telegram не повторял доставку. GET на тот же путь служит для проверки доступности URL при настройке webhook.

Безопасность

Пароли не хранятся в открытом виде; длина пароля для bcrypt ограничена 72 байтами. Коды одноразовые, с истечением по времени. Секрет JWT задаётся переменной SECRET_KEY в окружении production. При отсутствии TELEGRAM_BOT_TOKEN регистрация блокируется с ответом 503.

Связь с клиентом

На frontend AuthContext оборачивает вызовы authApi: register, verify, login, loginRequest, loginVerify, logout и загрузку профиля через GET /api/auth/me при наличии сохранённого токена.

Итог

Реализована цепочка «регистрация — привязка бота — код в Telegram — JWT», альтернативный вход с кодом и сброс пароля через тот же канал. Клиентская часть экранов входа и верификации описана в п. 3.3.
