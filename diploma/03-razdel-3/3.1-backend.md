3.1. Реализация серверной части

Серверная часть реализует спроектированный REST API (п. 2.3) и бизнес-логику бронирований. Проект размещён в каталоге backend; язык — Python 3.11+, фреймворк — FastAPI, доступ к данным — SQLAlchemy 2.0.

Структура проекта

Корень backend содержит зависимости (requirements.txt), конфигурацию Alembic, каталог tests и пакет app. Внутри app выделены слои, описанные при проектировании (п. 2.1): api (маршруты), models (ORM), schemas (Pydantic), services (доменная логика), core (безопасность и зависимости), а также config.py, database.py и main.py.

Миграции схемы БД хранятся в alembic/versions; каждая ревизия отражает эволюцию таблиц из п. 2.2.

Инициализация приложения

Файл main.py создаёт экземпляр FastAPI через фабрику create_app. При старте выполняется lifespan-хук: создаётся каталог для загрузок (uploads). Подключены middleware CORS с перечнем origins из настроек, роутеры с префиксами /api, /api/rooms, /api/bookings, /api/admin, статическая раздача файлов по пути /uploads и служебный маршрут GET /health.

Класс Settings (config.py) наследует pydantic-settings BaseSettings: параметры database_url, secret_key, срок жизни JWT, токен Telegram, SMTP и CORS читаются из .env или переменных окружения облака. Функция get_settings кэшируется, чтобы не перечитывать конфигурацию на каждый запрос.

Модуль database.py создаёт engine с pool_pre_ping, фабрику SessionLocal и генератор get_db для внедрения сессии в маршруты. Одна HTTP-обработка соответствует одной сессии; по завершении запроса сессия закрывается.

Модели данных

ORM-модели описаны в стиле SQLAlchemy 2.0 (Mapped, mapped_column, relationship). Центральные сущности — User, Room, Booking; вспомогательные — RoomPhoto (таблица room_images), RoomReview, таблицы одноразовых кодов и telegram_pending_links.

У модели Booking задан составной индекс ix_bookings_room_time по полям room_id, start_time, end_time для ускорения проверки пересечений. Внешние ключи на пользователя и комнату настроены с ON DELETE CASCADE.

Pydantic-схемы в schemas отделяют контракт API от таблиц: например, BookingCreate принимает room_id и интервал времени, BookingResponse возвращает данные с вложенным кратким описанием комнаты. Валидация типов и обязательных полей выполняется до входа в обработчик маршрута.

Реализация бронирований

Роутер bookings.py сосредоточивает правила предметной области. Константы в модуле задают минимальную длительность 30 минут, максимальную 6 часов, буфер 15 минут и дедлайн отмены 30 минут до начала.

При POST /api/bookings выполняется цепочка проверок: корректность интервала, запрет брони в прошлом, существование комнаты, затем вызов функции has_booking_conflict из services/booking_conflict.py. Алгоритм расширяет запрашиваемый интервал на буфер с обеих сторон и ищет в БД запись, у которой start_time < конец_проверки и end_time > начало_проверки. При нахождении конфликта возвращается HTTP 409.

Публичный GET /api/bookings с параметрами room_id, from_time, to_time отдаёт брони комнаты за период — используется календарём на клиенте. GET /api/bookings/me и операции с одной бронью требуют JWT; отмена реализована удалением строки (DELETE), с проверкой владельца и времени.

Управление комнатами

Роутер rooms.py обслуживает справочник: фильтрация списка по вместимости, текстовому поиску и полям адреса; загрузка фото через multipart с проверкой расширения и размера (до 5 МБ); сохранение относительного пути в room_images. Отзывы создаются авторизованным пользователем; средняя оценка вычисляется при формировании ответа о комнате.

Операции изменения справочника защищены зависимостью AdminUser: сначала извлекается пользователь по JWT, затем проверяется признак администратора, иначе — 403.

Безопасность и администрирование

Модуль core/security.py содержит хэширование паролей bcrypt и создание/разбор JWT (python-jose, HS256). Зависимости get_current_user и require_admin в core/dependencies.py используются во всех защищённых маршрутах.

Роутер admin.py реализует список администраторов, назначение по email (с автоматическим подтверждением учётной записи) и снятие роли с ограничениями «не снять себя» и «остаётся минимум один администратор».

Сценарии регистрации, входа с кодом в Telegram и обработка webhook вынесены в api/auth.py и api/telegram_webhook.py; их реализация рассматривается в п. 3.2.

Тестирование серверной логики

В каталоге tests размещены модульные тесты Pytest: проверка has_booking_conflict на граничных случаях буфера и интеграционные вызовы API бронирований через httpx ASGI-клиент. Для изоляции может использоваться SQLite вместо PostgreSQL (настройка в conftest.py).

Итог

Серверная часть обеспечивает REST-интерфейс, персистентность данных и автоматические проверки при бронировании. Детали аутентификации и Telegram — в п. 3.2; клиентское приложение — в п. 3.3.
