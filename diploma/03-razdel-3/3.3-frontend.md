3.3. Реализация клиентской части

Клиентское приложение реализует спроектированный интерфейс (п. 2.4) и взаимодействует с REST API (п. 2.3). Проект собран на React 18 и TypeScript, сборка — Vite; стили — Tailwind CSS.

Структура исходного кода

Каталог src разделён по назначению. В api расположены модули client, auth, rooms, bookings, admin и profile — тонкие обёртки над fetch. Компоненты Layout, SkeletonBlocks и EmptyStateCard используются на нескольких экранах. Каталог pages содержит экраны, перечисленные в табл. 2.6. Контекст AuthContext и хук i18n обеспечивают сессию и локализацию. Хуки useFavoriteRooms и useRecentRooms работают с localStorage без отдельного API.

Точка входа main.tsx монтирует приложение в провайдеры AuthProvider и I18nProvider; маршрутизация задана в App.tsx.

HTTP-клиент

Модуль api/client.ts реализует функцию request: подставляет JWT из localStorage в заголовок Authorization, сериализует JSON, разбирает ошибки FastAPI в тип ApiError.

Функция buildApiUrl формирует абсолютный адрес запроса с учётом VITE_API_URL (на Render — URL backend). В режиме разработки при пустом VITE_API_URL используется origin dev-сервера и прокси Vite на /api. Функция mediaUrl строит URL фотографий для тега img, если статика отдаётся с другого домена.

Аутентификация на клиенте

AuthContext при загрузке проверяет access_token в localStorage и запрашивает GET /api/auth/me. Методы register, login, loginRequest и loginVerify вызывают соответствующие эндпоинты и сохраняют токен после успеха. logout удаляет токен и сбрасывает состояние пользователя.

Страницы RegisterPage и VerifyEmailPage реализуют цепочку регистрации: форма, переход по ссылке на бота, ввод кода. LoginPage поддерживает обычный вход и вариант с кодом в Telegram. ForgotPasswordPage и ResetPasswordPage — восстановление доступа.

Защита маршрутов

Компоненты PrivateRoute и AdminRoute в App.tsx оборачивают страницы профиля, бронирований и админ-панели. Пока идёт проверка сессии, показывается индикатор загрузки; при отсутствии прав выполняется Navigate на /login или главную.

Ключевые экраны

HomePage для авторизованного пользователя загружает ближайшие бронирования и блоки избранного и истории; для гостя — лендинг.

RoomListPage запрашивает GET /api/rooms с параметрами фильтров из формы и отображает карточки с переходом на детальную страницу.

RoomDetailPage — центральный экран бронирования. Загружаются данные комнаты, отзывы и брони за видимый период календаря. Компонент FullCalendar (плагины dayGrid и timeGrid) показывает занятые интервалы как события. Форма отправляет POST /api/bookings; при ответе 409 пользователю выводится текст ошибки с сервера. Реализованы избранное, галерея фото и отправка отзыва.

MyBookingsPage и BookingDetailsPage отображают список и детали собственных броней; отмена вызывает DELETE с подтверждением и проверкой дедлайна на клиенте.

AdminPage разделена на вкладки управления комнатами (форма, загрузка файлов через multipart) и администраторами (promote/demote через admin API).

ProfilePage позволяет редактировать поля профиля через PATCH /api/users/me.

Удобство использования

Длительные запросы сопровождаются SkeletonBlocks; пустые списки — EmptyStateCard. Переводы вынесены в i18n/translations.ts, переключатель языка — в Layout. Вёрстка на utility-классах Tailwind адаптируется к ширине экрана.

Итог

Клиентская часть замыкает цепочку «интерфейс — API — база данных», реализует все основные сценарии пользователя и администратора. Автоматическое тестирование и развёртывание рассматриваются в п. 3.4 и 3.5.
