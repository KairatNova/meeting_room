3.4. Тестирование системы

Проверка качества выполнена на двух уровнях: модульные тесты изолированной логики и интеграционные тесты HTTP API. На клиенте — модульные тесты утилит и компонентов. Ручная проверка сценариев в браузере дополняет автоматизацию при приёмке прототипа.

Подход и инструменты

Backend: фреймворк Pytest, клиент TestClient из Starlette для вызова ASGI-приложения без отдельного процесса. Для тестов БД поднимается in-memory SQLite с переопределением зависимости get_db — production PostgreSQL не затрагивается.

Frontend: Vitest и среда happy-dom; для компонентов — React Testing Library. Запуск: pytest в каталоге backend, npm test в каталоге frontend.

Критерии полноты тестирования

Приоритет в тест-плане отдан не количеству кейсов, а рисковым точкам предметной логики. Для backend это правила пересечения интервалов и ограничения отмены, для frontend — корректная обработка авторизации и ошибок API. Такой подход позволяет быстро выявлять регрессии, которые напрямую влияют на бизнес-сценарии, и не перегружать проект второстепенными тестами.

В качестве минимального критерия приёмки использовалось требование: любой критичный пользовательский сценарий должен иметь хотя бы одну автоматическую проверку на уровне API или клиентской логики. Это обеспечивает баланс между скоростью разработки и контролем качества.

Тестирование серверной логики

Модуль tests/test_booking_conflict.py проверяет функцию has_booking_conflict: отсутствие броней, пересечение интервалов, касание границ без конфликта, учёт 15-минутного буфера, игнорирование другой комнаты, исключение редактируемой брони из проверки.

Интеграционные тесты tests/test_bookings_api.py покрывают ключевые сценарии API бронирований (табл. 3.1).

Таблица 3.1 – Основные автотесты API бронирований

| Проверка | Ожидаемый результат |
| --- | --- |
| Создание брони в свободный слот | HTTP 201 |
| Пересечение с существующей бронью | HTTP 409 |
| Длительность более 6 часов | HTTP 400 |
| Нарушение буфера 15 минут между бронями | HTTP 409 |
| Отмена менее чем за 30 минут до начала | HTTP 400 |
| Просмотр своей брони | HTTP 200 |
| Просмотр чужой брони | HTTP 403 |

Тесты подтверждают соответствие реализации бизнес-правилам, сформулированным в п. 1.3.

Тестирование клиентской части

Файл api/client.test.ts проверяет добавление заголовка Authorization из localStorage, разбор тела ошибки ApiError при статусе 409 и формирование URL через buildApiUrl.

Тест HomePage.test.tsx с моками API и AuthContext проверяет отображение пустого состояния при отсутствии комнат.

Полный сквозной UI-тест (E2E) в проекте не выделен; критичные сценарии регистрации, бронирования и админ-панели рекомендуется прогонять вручную на стенде после деплоя.

Ручная приёмка после деплоя

После выкладки на облачный стенд выполняется короткий smoke-набор: регистрация нового пользователя, подтверждение через Telegram, создание и отмена брони, проверка конфликта 409, вход администратора и изменение карточки комнаты. Эти шаги занимают ограниченное время, но позволяют поймать ошибки окружения, которые не проявляются в локальном тестовом контуре.

Результаты ручной проверки фиксируются чек-листом с датой, версией приложения и кратким статусом каждого сценария. Такой формат удобен для подготовки к защите: можно показать не только факт запуска тестов, но и воспроизводимую процедуру приёмки.

Результаты

При локальном запуске набор тестов backend и frontend завершается успешно, что подтверждает корректность проверки конфликтов, кодов ответа API и базовой работы HTTP-клиента. Регрессии при изменении booking_conflict или маршрутов bookings выявляются до выкладки в облако.
