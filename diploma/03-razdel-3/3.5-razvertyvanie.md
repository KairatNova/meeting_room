3.5. Развёртывание приложения

Разработанный комплекс размещается в облаке без собственного сервера. В репозитории подготовлены конфигурации для двух PaaS-платформ: основной вариант — Render, альтернативный — Railway. Оба поддерживают managed PostgreSQL, отдельные сервисы backend и frontend и запуск миграций при старте API.

Развёртывание на Render

Используется декларативный файл render.yaml в корне репозитория. Blueprint создаёт три ресурса: база meeting-room-db (PostgreSQL), web-сервис meeting-room-backend (каталог backend) и статический сайт meeting-room-frontend (сборка dist).

Команда запуска backend: alembic upgrade head, затем Uvicorn на порту PORT. Проверка доступности — GET /health. Для SPA настроен rewrite всех путей на index.html.

Переменные backend: DATABASE_URL (из базы), SECRET_KEY, DEBUG=false, CORS_ORIGINS с URL frontend, UPLOAD_DIR=uploads, TELEGRAM_BOT_TOKEN и TELEGRAM_BOT_USERNAME. На frontend при сборке задаётся VITE_API_URL — публичный адрес backend без суффикса /api. Значения CORS_ORIGINS и VITE_API_URL должны совпадать с реальными доменами после деплоя.

На бесплатном тарифе каталог uploads эфемерен: фотографии могут пропадать после перезапуска контейнера. Для учебного стенда это допустимо; для постоянного хранения нужен том или внешнее объектное хранилище.

Развёртывание на Railway (альтернатива)

Проект — монорепозиторий с двумя приложениями. В Railway создаются два сервиса из одного репозитория.

Сервис backend: корневая директория backend, конфигурация backend/railway.json (сборщик Nixpacks). Команда старта совпадает с Render — миграции Alembic и Uvicorn. Подключается плагин PostgreSQL; в переменные передаются DATABASE_URL, SECRET_KEY, CORS_ORIGINS, параметры Telegram.

Сервис frontend: корневая директория frontend, файл frontend/railway.json. После npm run build приложение отдаётся через npm run preview на порту PORT. Задаётся VITE_API_URL с адресом backend-сервиса.

Преимущества Railway: привычная панель, быстрый деплой из GitHub, единая схема «два сервиса + Postgres» для учебных проектов. Render удобен единым Blueprint-файлом и отдельным static-хостингом без Node-процесса на frontend. В работе зафиксирован рабочий стенд на Render; конфигурация Railway сохранена в репозитории для переноса или сравнения платформ (подробности — в deploy/DEPLOY_RENDER.md и deploy/DEPLOY_RAILWAY.md).

Общие шаги после выкладки

1. Настроить webhook Telegram: POST https://ДОМЕН_BACKEND/api/telegram/webhook.
2. Проверить логи применения миграций.
3. Открыть URL frontend, пройти регистрацию, бронирование и вход администратора.

Локальная разработка

Backend: виртуальное окружение Python, зависимости из requirements.txt, .env с DATABASE_URL на локальный PostgreSQL, alembic upgrade head, uvicorn app.main:app. Frontend: npm install, npm run dev; прокси Vite перенаправляет /api на localhost:8000.
