Приложения

Материалы приложений оформляются после списка источников, с новой страницы. Каждое приложение начинается с заголовка по центру: «Приложение А», «Приложение Б» и т.д., затем название. Листинги — шрифт Courier New 12 пт (или по методичке), скриншоты — с подписью под рисунком.

Приложение А

ER-диаграмма базы данных

На рисунке А.1 представлена логическая схема базы данных разрабатываемой системы: таблицы users, rooms, room_images, bookings, room_reviews, email_verification_codes, login_verification_codes, password_reset_codes, telegram_pending_links и связи между ними.

Диаграмму подготовить в dbdiagram.io по коду из файла DIPLOMA_DB_DIAGRAM.md в корне репозитория проекта, экспортировать в PNG или PDF и вставить в документ Word.

Приложение Б

Листинги фрагментов программного кода

В приложение включаются фрагменты, иллюстрирующие ключевую логику (полные файлы — в репозитории проекта meeting_room).

Б.1. Проверка конфликтов бронирований с учётом буфера — файл backend/app/services/booking_conflict.py, функция has_booking_conflict.

Б.2. Создание и отмена бронирования — файл backend/app/api/bookings.py, функции create_booking и cancel_booking.

Б.3. Обработка команды /start в Telegram — файл backend/app/api/telegram_webhook.py, функция telegram_webhook (фрагмент привязки chat_id и отправки кода).

Б.4. Контекст авторизации на клиенте — файл frontend/src/context/AuthContext.tsx (хранение токена, загрузка профиля).

Б.5. Календарь занятости на странице комнаты — файл frontend/src/pages/RoomDetailPage.tsx (подключение FullCalendar и загрузка броней).

Готовые листинги с подсветкой синтаксиса можно взять из файла DIPLOMA_APPENDIX.md в корне репозитория.

Приложение В

Скриншоты пользовательского интерфейса

Скриншоты выполняются на развёрнутом стенде (Render) или локально. Рекомендуемый порядок:

В.1. Главная страница — дашборд авторизованного пользователя.

В.2. Каталог переговорных комнат с фильтрами.

В.3. Карточка комнаты с календарём занятости (FullCalendar).

В.4. Сообщение об ошибке при конфликте бронирования (HTTP 409).

В.5. Страница «Мои бронирования».

В.6. Детали бронирования и отмена.

В.7. Административная панель — управление комнатами.

В.8. Административная панель — назначение администраторов.

В.9. Страницы регистрации и подтверждения кода (/verify-email).

В.10. Интерактивная документация API — Swagger UI по адресу /docs.

Подписи: Рисунок В.1 – …, Рисунок В.2 – … (нумерация сквозная в пределах приложения или работы — по методичке кафедры).

Приложение Г

Фрагменты конфигурации развёртывания

Г.1. Файл render.yaml (Blueprint платформы Render) — декларация PostgreSQL, backend и static frontend.

Г.2. Пример переменных окружения backend — файл backend/.env.example (DATABASE_URL, SECRET_KEY, CORS_ORIGINS, TELEGRAM_BOT_TOKEN).

Г.3. Пример переменных окружения frontend — файл frontend/.env.example (VITE_API_URL).

Г.4. Конфигурация альтернативного деплоя — файлы backend/railway.json и frontend/railway.json (платформа Railway).

Актуальное содержимое файлов — в репозитории проекта; при вставке в Word укажите имя файла в подписи под листингом.
