1.3. Постановка задачи и требования к системе

По результатам анализа предметной области (п. 1.1) и сравнения аналогов (п. 1.2) сформулирована задача разработки: создать веб-приложение, которое заменяет ручной учёт переговорных комнат единым сервисом с проверкой конфликтов, ролевым доступом и подтверждением личности через Telegram. Ниже зафиксированы границы системы, функциональные и нефункциональные требования, бизнес-правила и сценарии использования. Формулировки согласованы с реализуемым прототипом и служат основой для проектирования (раздел 2) и приёмочной проверки (раздел 3).

Границы и допущения

В scope входят: веб-клиент (SPA), сервер REST API, реляционная база данных, интеграция с Telegram Bot API, хранение фотографий комнат на сервере приложения, развёртывание в облаке.

Вне scope на текущем этапе: нативные мобильные приложения, интеграция с Google Calendar и Microsoft 365, оплата бронирований, SMS- и email-рассылки (почта используется только как уникальный идентификатор учётной записи, без отправки писем), оборудование «умных замков» и датчики присутствия в комнатах.

Допущения: у пользователя есть аккаунт в Telegram для получения кодов; администратор создаётся вручную в базе или назначается из существующих пользователей; все временные метки бронирований на сервере хранятся в UTC и отображаются клиентом с учётом локального часового пояса браузера.

Роли и права доступа

Система оперирует двумя основными ролями.

Гость (неавторизованный посетитель) может просматривать список комнат, карточку комнаты, публичные отзывы и календарь занятости (для оценки свободных слотов). Создание брони, управление профилем и администрирование недоступны.

Пользователь (авторизованный, is_verified = true) создаёт и отменяет собственные бронирования, редактирует профиль, оставляет отзывы о комнатах, использует избранное и историю просмотров на клиенте.

Администратор (is_admin = true) обладает всеми правами пользователя и дополнительно управляет справочником комнат (включая загрузку и удаление фотографий), назначает и снимает других администраторов в рамках ограничений, описанных в ФТ-4.

Функциональные требования

Требования сгруппированы по подсистемам. Для каждого указан приоритет: О — обязательный, Ж — желательный (реализован в прототипе частично или на стороне клиента без серверного API).

ФТ-1. Управление учётными записями и аутентификация

ФТ-1.1 (О). Регистрация: имя, email (уникальный логин), пароль (не менее 8 символов), идентификатор Telegram (@username или номер телефона в международном формате).

ФТ-1.2 (О). После регистрации пользователю выдаётся ссылка на Telegram-бота; по команде Start (и при необходимости передаче контакта телефона) бот связывает chat_id с учётной записью и доставляет шестизначный код подтверждения.

ФТ-1.3 (О). Подтверждение регистрации вводом кода на странице верификации; при успехе устанавливается is_verified = true и выдаётся JWT.

ФТ-1.4 (О). Вход по email и паролю после подтверждения аккаунта.

ФТ-1.5 (О). Альтернативный вход: запрос кода (login-request) по email, Telegram-нику или телефону с проверкой пароля, доставка кода в Telegram, подтверждение (login-verify) и выдача JWT.

ФТ-1.6 (О). Восстановление пароля: запрос кода в Telegram (forgot-password), установка нового пароля по коду (reset-password).

ФТ-1.7 (О). Получение данных текущего пользователя (GET /auth/me) по JWT; обновление полей профиля (PATCH /users/me).

ФТ-1.8 (Ж). Многоязычный интерфейс клиента (русский и дополнительные локали в файле переводов).

ФТ-2. Управление переговорными комнатами

ФТ-2.1 (О). Публичный просмотр списка комнат без авторизации.

ФТ-2.2 (О). Фильтрация и сортировка списка: минимальная вместимость, поиск по названию и описанию, удобства, регион, город, район, адрес; сортировка по имени, вместимости, дате добавления.

ФТ-2.3 (О). Просмотр детальной карточки комнаты: параметры, фотографии, средняя оценка по отзывам.

ФТ-2.4 (О). Создание, изменение и удаление комнаты — только администратор.

ФТ-2.5 (О). Загрузка изображений комнаты (форматы JPEG, PNG, GIF, WebP; ограничение размера файла на сервере) и удаление отдельных фотографий — только администратор.

ФТ-2.6 (О). Просмотр отзывов по комнате; добавление отзыва (текст, оценка) — авторизованный пользователь.

ФТ-2.7 (Ж). Избранные комнаты и недавно просмотренные — хранение идентификаторов в localStorage браузера без отдельного API.

ФТ-3. Управление бронированиями

ФТ-3.1 (О). Создание бронирования: комната, время начала и окончания; доступно только верифицированному пользователю.

ФТ-3.2 (О). Просмотр бронирований комнаты за период (для календаря на карточке комнаты) с фильтром room_id, from_time, to_time.

ФТ-3.3 (О). Просмотр списка собственных бронирований (/bookings/me) и деталей одной брони.

ФТ-3.4 (О). Отмена собственной будущей брони с учётом дедлайна (см. БП-4).

ФТ-3.5 (О). Отклонение создания брони при пересечении с существующей активной бронью с учётом буфера (HTTP 409 Conflict).

ФТ-3.6 (О). Запрет доступа к чужой брони (HTTP 403), кроме сценариев администратора при расширении в будущем.

ФТ-4. Администрирование

ФТ-4.1 (О). Просмотр списка пользователей с ролью администратора.

ФТ-4.2 (О). Назначение администратора по email зарегистрированного пользователя; при назначении аккаунт автоматически помечается как подтверждённый.

ФТ-4.3 (О). Снятие роли администратора с другого пользователя; запрет снятия с себя; в системе всегда остаётся не менее одного администратора.

ФТ-4.4 (О). Проверка прав администратора на клиенте (AdminRoute) и на сервере (зависимость AdminUser).

ФТ-5. Пользовательский интерфейс

ФТ-5.1 (О). Одностраничное приложение с маршрутизацией: главная (дашборд / лендинг), каталог комнат, карточка комнаты с календарём FullCalendar, мои бронирования, профиль, админ-панель, страницы входа, регистрации, верификации, восстановления пароля.

ФТ-5.2 (О). Защищённые маршруты: перенаправление неавторизованного пользователя на страницу входа.

ФТ-5.3 (Ж). Состояния загрузки (скелетоны), пустые состояния, индикация отправки форм.

ФТ-6. Документирование и эксплуатация API

ФТ-6.1 (О). Автоматическая генерация описания REST API (Swagger UI по адресу /docs, ReDoc по /redoc).

ФТ-6.2 (О). Развёртывание комплекса frontend + backend + PostgreSQL по декларативной конфигурации облачной платформы.

UC-1. Регистрация и подтверждение через Telegram

Актор: гость. Предусловие: Telegram-бот настроен, сервер доступен.

Основной поток:

1. Гость заполняет форму регистрации (имя, email, пароль, Telegram).
2. Сервер создаёт запись пользователя с is_verified = false, генерирует код и ссылку на бота.
3. Гость открывает бота, нажимает Start; при регистрации по телефону — передаёт контакт.
4. Бот отправляет шестизначный код в Telegram.
5. Гость вводит email и код на странице верификации.
6. Сервер устанавливает is_verified = true, возвращает JWT.

Альтернативы: неверный или просроченный код — сообщение об ошибке, повторная регистрация; дубликат email у подтверждённого пользователя — отказ.

UC-6. Создание бронирования

Актор: верифицированный пользователь. Предусловие: пользователь авторизован, комната активна.

Основной поток:

1. Пользователь открывает карточку комнаты, изучает календарь занятости.
2. Указывает дату и время начала и окончания.
3. Клиент отправляет POST /api/bookings.
4. Сервер проверяет БП-1–БП-5; при успехе сохраняет бронь и возвращает 201.

Альтернатива 4а: конфликт — UC-7, пользователь выбирает другой слот.

UC-9. Управление комнатами

Актор: администратор. Предусловие: JWT с is_admin = true.

Основной поток: администратор открывает панель, создаёт или редактирует комнату, загружает фотографии, сохраняет изменения через API. При удалении комнаты связанные данные обрабатываются согласно настройкам внешних ключей в БД.

Связь требований с разделами работы

Функциональные требования ФТ-1–ФТ-4 и бизнес-правила БП-1–БП-10 реализуются в разделе 3 (backend, Telegram, frontend). Проектные артефакты — модель данных, контракт API, макеты экранов — формируются в разделе 2. Обоснование выбора FastAPI, PostgreSQL, React и сопутствующих библиотек приводится в п. 1.4.

Принятые требования закрывают пробелы, выявленные при сравнении с аналогами: единое расписание, проверка конфликтов и буфера, русскоязычный интерфейс, верификация через Telegram, возможность развёртывания у заказчика без привязки к конкретному коворкингу.
