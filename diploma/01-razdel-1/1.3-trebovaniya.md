1.3. Постановка задачи и требования к программной системе

По результатам анализа предметной области (п. 1.1) и сравнения аналогов (п. 1.2) сформулирована задача разработки: создать веб-приложение, которое заменяет ручной учёт переговорных комнат единым сервисом с проверкой конфликтов, ролевым доступом и подтверждением личности через Telegram. Ниже зафиксированы границы системы, функциональные и нефункциональные требования, бизнес-правила и сценарии использования. Формулировки согласованы с реализуемым прототипом и служат основой для проектирования (раздел 2) и приёмочной проверки (раздел 3).

Границы и допущения

В состав разработки входят: веб-клиент (SPA), сервер REST API, реляционная база данных, интеграция с Telegram Bot API, хранение фотографий комнат на сервере приложения, развёртывание в облаке.

На текущем этапе не рассматриваются: нативные мобильные приложения, интеграция с Google Calendar и Microsoft 365, оплата бронирований, SMS- и email-рассылки (почта используется только как уникальный идентификатор учётной записи, без отправки писем), оборудование «умных замков» и датчики присутствия в комнатах.

Допущения: у пользователя есть аккаунт в Telegram для получения кодов; администратор создаётся вручную в базе или назначается из существующих пользователей; все временные метки бронирований на сервере хранятся в UTC и отображаются клиентом с учётом локального часового пояса браузера.

Роли и права доступа

Система оперирует двумя основными ролями.

Гость (неавторизованный посетитель) может просматривать список комнат, карточку комнаты, публичные отзывы и календарь занятости (для оценки свободных слотов). Создание брони, управление профилем и администрирование недоступны.

Пользователь (авторизованный, с подтверждённой учётной записью) создаёт и отменяет собственные бронирования, редактирует профиль, оставляет отзывы о комнатах, использует избранное и историю просмотров на клиенте.

Администратор обладает всеми правами пользователя и дополнительно управляет справочником комнат (включая загрузку и удаление фотографий), назначает и снимает других администраторов в рамках ограничений, описанных в блоке «Администрирование».

Функциональные требования

Требования сгруппированы по подсистемам. Для каждого указан приоритет: обязательный или желательный (реализован в прототипе частично или на стороне клиента без серверного API).

Управление учётными записями и аутентификация

Обязательно: регистрация с указанием имени, email (уникальный логин), пароля (не менее 8 символов) и идентификатора Telegram (@username или номер телефона в международном формате).

Обязательно: после регистрации пользователю выдаётся ссылка на Telegram-бота; по команде Start (и при необходимости передаче контакта телефона) бот связывает chat_id с учётной записью и доставляет шестизначный код подтверждения.

Обязательно: подтверждение регистрации вводом кода на странице верификации; при успехе учётная запись помечается как подтверждённая, выдаётся JWT.

Обязательно: вход по email и паролю после подтверждения аккаунта.

Обязательно: альтернативный вход — запрос кода по email, Telegram-нику или телефону с проверкой пароля, доставка кода в Telegram, подтверждение и выдача JWT.

Обязательно: восстановление пароля — запрос кода в Telegram, установка нового пароля по коду.

Обязательно: получение данных текущего пользователя по JWT; обновление полей профиля.

Желательно: многоязычный интерфейс клиента (русский и дополнительные локали в файле переводов).

Управление переговорными комнатами

Обязательно: публичный просмотр списка комнат без авторизации.

Обязательно: фильтрация и сортировка списка — минимальная вместимость, поиск по названию и описанию, удобства, регион, город, район, адрес; сортировка по имени, вместимости, дате добавления.

Обязательно: просмотр детальной карточки комнаты — параметры, фотографии, средняя оценка по отзывам.

Обязательно: создание, изменение и удаление комнаты — только администратор.

Обязательно: загрузка изображений комнаты (форматы JPEG, PNG, GIF, WebP; ограничение размера файла на сервере) и удаление отдельных фотографий — только администратор.

Обязательно: просмотр отзывов по комнате; добавление отзыва (текст, оценка) — авторизованный пользователь.

Желательно: избранные комнаты и недавно просмотренные — хранение идентификаторов в localStorage браузера без отдельного API.

Управление бронированиями

Обязательно: создание бронирования (комната, время начала и окончания) — только подтверждённый пользователь.

Обязательно: просмотр бронирований комнаты за период (для календаря на карточке комнаты) с фильтром по комнате и интервалу времени.

Обязательно: просмотр списка собственных бронирований и деталей одной брони.

Обязательно: отмена собственной будущей брони с учётом дедлайна отмены (см. бизнес-правила).

Обязательно: отклонение создания брони при пересечении с существующей активной бронью с учётом буфера (HTTP 409 Conflict).

Обязательно: запрет доступа к чужой брони (HTTP 403), кроме сценариев администратора при расширении в будущем.

Администрирование

Обязательно: просмотр списка пользователей с ролью администратора.

Обязательно: назначение администратора по email зарегистрированного пользователя; при назначении аккаунт автоматически помечается как подтверждённый.

Обязательно: снятие роли администратора с другого пользователя; запрет снятия с себя; в системе всегда остаётся не менее одного администратора.

Обязательно: проверка прав администратора на клиенте и на сервере.

Пользовательский интерфейс

Обязательно: одностраничное приложение с маршрутизацией — главная (дашборд / лендинг), каталог комнат, карточка комнаты с календарём FullCalendar, мои бронирования, профиль, админ-панель, страницы входа, регистрации, верификации, восстановления пароля.

Обязательно: защищённые маршруты — перенаправление неавторизованного пользователя на страницу входа.

Желательно: состояния загрузки (скелетоны), пустые состояния, индикация отправки форм.

Документирование и эксплуатация API

Обязательно: автоматическая генерация описания REST API (Swagger UI по адресу /docs, ReDoc по /redoc).

Обязательно: развёртывание комплекса frontend + backend + PostgreSQL по декларативной конфигурации облачной платформы.

Регистрация и подтверждение через Telegram

Актор: гость. Предусловие: Telegram-бот настроен, сервер доступен.

Основной поток:

1. Гость заполняет форму регистрации (имя, email, пароль, Telegram).
2. Сервер создаёт запись пользователя с неподтверждённым статусом, генерирует код и ссылку на бота.
3. Гость открывает бота, нажимает Start; при регистрации по телефону — передаёт контакт.
4. Бот отправляет шестизначный код в Telegram.
5. Гость вводит email и код на странице верификации.
6. Сервер помечает учётную запись как подтверждённую, возвращает JWT.

Альтернативы: неверный или просроченный код — сообщение об ошибке, повторная регистрация; дубликат email у подтверждённого пользователя — отказ.

Создание бронирования

Актор: подтверждённый пользователь. Предусловие: пользователь авторизован, комната активна.

Основной поток:

1. Пользователь открывает карточку комнаты, изучает календарь занятости.
2. Указывает дату и время начала и окончания.
3. Клиент отправляет запрос на создание бронирования.
4. Сервер проверяет бизнес-правила длительности, буфера и пересечений; при успехе сохраняет бронь и возвращает 201.

Альтернатива на шаге 4: конфликт расписания — отклонение с кодом 409, пользователь выбирает другой слот.

Управление комнатами

Актор: администратор. Предусловие: JWT с правами администратора.

Основной поток: администратор открывает панель, создаёт или редактирует комнату, загружает фотографии, сохраняет изменения через API. При удалении комнаты связанные данные обрабатываются согласно настройкам внешних ключей в БД.

Связь требований с разделами работы

Функциональные требования к учётным записям, комнатам, бронированиям и администрированию, а также бизнес-правила из табл. 1.3 реализуются в разделе 3 (серверная часть, Telegram, клиент). Проектные артефакты — модель данных, контракт API, макеты экранов — формируются в разделе 2. Обоснование выбора FastAPI, PostgreSQL, React и сопутствующих библиотек приводится в п. 1.4.

Принятые требования закрывают пробелы, выявленные при сравнении с аналогами: единое расписание, проверка конфликтов и буфера, русскоязычный интерфейс, верификация через Telegram, возможность развёртывания у заказчика без привязки к конкретному коворкингу.
