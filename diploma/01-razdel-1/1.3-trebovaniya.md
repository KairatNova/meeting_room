1.3. Постановка задачи и требования к программной системе

Задача разработки: веб-приложение вместо ручного учёта переговорных — единое расписание, проверка конфликтов, роли, подтверждение личности через Telegram. Ниже — границы системы, требования и правила; детали API и реализации — в разделах 2 и 3.

Границы системы

В scope: SPA-клиент, REST API (FastAPI), PostgreSQL, Telegram Bot API, хранение фото комнат, облачное развёртывание.

Вне scope: мобильные нативные клиенты, интеграция с Google Calendar / Microsoft 365, оплата, email-рассылка кодов (email — только логин), «умные замки».

Допущения: у пользователя есть Telegram; администратор назначается вручную или через API; время броней на сервере — UTC.

Роли

Гость просматривает комнаты, отзывы и календарь занятости. Подтверждённый пользователь бронирует и отменяет свои слоты, редактирует профиль, пишет отзывы; избранное и «недавние» — в localStorage браузера. Администратор ведёт справочник комнат и фото, назначает и снимает администраторов (не себя; в системе остаётся минимум один админ).
