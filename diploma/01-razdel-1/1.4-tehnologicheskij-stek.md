1.4. Выбор и обоснование технологического стека

Требования п. 1.3 задают функциональность и качество системы; технологический стек определяет, какими средствами они реализуются. При выборе компонентов использовались критерии, согласованные с целями выпускной квалификационной работы:

1) соответствие модели «клиент — REST API — СУБД»;

2) зрелость экосистемы и доступность документации;

3) скорость разработки и сопровождения одним разработчиком;

4) отсутствие обязательной платы за лицензии ПО;

5) возможность бесплатного или недорогого облачного развёртывания;

6) поддержка типизации, автоматических тестов и генерации описания API.

Ниже обосновывается выбор по уровням архитектуры. Итоговая сводка приведена в табл. 1.6.

Язык и среда серверной разработки

Серверная часть реализована на Python 3.11 и выше. Язык широко применяется в веб-разработке и анализе данных, имеет читаемый синтаксис и богатую стандартную библиотеку [Источник №]. Для задачи дипломного проекта важны библиотеки валидации (Pydantic), ORM (SQLAlchemy) и фреймворки ASGI. Альтернатива — Node.js (TypeScript на сервере): единый язык с frontend, но дублирование стека не даёт решающего выигрыша при уже выбранном React на клиенте, а экосистема Telegram и почтовых интеграций на Python для автора проекта используется проще.

Java (Spring Boot) и C# (ASP.NET Core) обеспечивают промышленную надёжность, однако требуют больше шаблонного кода и времени на конфигурацию для прототипа среднего масштаба. PHP (Laravel) остаётся распространённым, но для строгой типизации моделей и автогенерации OpenAPI в связке с современным SPA удобнее Python + FastAPI.

Фреймворк REST API: FastAPI

Для HTTP API выбран FastAPI (версия 0.109 и выше в зависимостях проекта). Фреймворк построен на Starlette и Pydantic 2: описания маршрутов сопровождаются аннотациями типов, тела запросов и ответов проверяются автоматически, ошибки валидации возвращаются в формате JSON с кодом 422.

Преимущества для данной работы:

— автоматическая публикация спецификации OpenAPI и интерфейсов Swagger UI (/docs) и ReDoc (/redoc), что закрывает НФТ-7 без ручного ведения документации;

— зависимости FastAPI (Depends) для внедрения сессии БД и текущего пользователя по JWT;

— запуск через Uvicorn как ASGI-сервер, при необходимости — асинхронные обработчики.

Сравнение с альтернативами.

Django и Django REST Framework (DRF) — полноценная платформа с админ-панелью, ORM и миграциями «из коробки». Для узкоспециализированного API бронирований админка Django дублировала бы собственную React-панель; объём настроек (settings, apps, serializers) выше, чем у лаконичного FastAPI-приложения с явной структурой каталогов api, models, schemas, services.

Flask — микрофреймворк без встроенной валидации и схем; пришлось бы подключать marshmallow или apiflask, вручную описывать OpenAPI. FastAPI сокращает количество служебного кода.

Express (Node.js) — гибкий, но типизация и валидация (zod, class-validator) настраиваются отдельно; дублирование языка с frontend не обязательно преимущество при разделении командных ролей «backend / frontend» в учебном проекте.

Вывод: FastAPI оптимален по соотношению объёма кода, качества API-контракта и скорости разработки для REST-сервиса средней сложности.

СУБД: PostgreSQL

В качестве системы управления базами данных выбран PostgreSQL (на стенде разработки и в облаке Render — версия 16 или совместимая). Реляционная модель естественно отражает связи «пользователь — бронирование — комната», требования ACID важны при одновременном создании броней.

PostgreSQL поддерживает транзакции, внешние ключи, индексы по временным полям (для выборок бронирований по room_id и интервалу), тип BOOLEAN и TIMESTAMP WITH TIME ZONE. СУБД бесплатна, развёртывается локально и как managed-сервис в облаке.

Альтернативы.

SQLite — удобна для модульных тестов (в проекте применяется в Pytest), но для многопользовательской записи в production слабее по блокировкам и не используется как основная БД в развёртывании.

MySQL / MariaDB — сопоставимы по возможностям; выбор PostgreSQL обусловлен привычной связкой с SQLAlchemy 2, наличием бесплатного тарифа на Render и строгой проверкой типов.

MongoDB — документная модель избыточна для нормализованной схемы из пяти связанных таблиц; отчёты по занятости проще выражать SQL-запросами.

Доступ к данным: SQLAlchemy 2 и Alembic

SQLAlchemy 2.0 (в requirements: sqlalchemy>=2.0.25) используется как ORM и конструктор запросов. Модели наследуют Declarative Base; связи задаются через relationship. Стиль 2.0 с единым Session и select() согласуется с рекомендациями документации и упрощает тестирование.

Alembic (>=1.13) версионирует схему БД: каждое изменение таблиц оформляется миграцией в каталоге alembic/versions, что позволяет воспроизводить базу на новом сервере командой upgrade head.

Альтернатива — сырые SQL-скрипты без миграций: быстрее на старте, но рискованно при командной работе и деплое. Альтернатива — Django ORM без Alembic возможна только внутри Django; проект на FastAPI требует отдельного инструмента миграций, и Alembic является отраслевым стандартом для SQLAlchemy.

Драйвер psycopg2-binary обеспечивает подключение к PostgreSQL из Python.

Аутентификация и безопасность на сервере

Пароли не хранятся в открытом виде. Применяется библиотека bcrypt: функции hash_password и verify_password в модуле security с ограничением длины пароля 72 байта в соответствии с ограничениями bcrypt.

Для выдачи и проверки сессий без состояния на сервере используются JWT (библиотека python-jose, алгоритм HS256, секрет SECRET_KEY из переменных окружения). Токен передаётся клиентом в заголовке Authorization: Bearer. Срок жизни токена задаётся в настройках. Такой подход соответствует НФТ-1 и типичной практике SPA [Источник №].

Альтернатива — серверные сессии в Redis: проще отзыв токенов, но для дипломного прототипа усложняет инфраструктуру (дополнительный сервис). OAuth2 через внешние провайдеры не требуется по п. 1.3.

Интеграция с Telegram Bot API

Доставка кодов регистрации, входа и сброса пароля реализована через HTTPS-запросы к api.telegram.org и webhook POST /api/telegram/webhook для входящих сообщений бота. Выбор обусловлен требованиями ФТ-1 и результатами анализа аналогов (п. 1.2): в СНГ Telegram распространён как канал уведомлений.

Альтернатива — SMTP и email-рассылка: требует настройки почтового сервера, SPF/DKIM, чаще попадает в спам. SMS — платно. Собственное мобильное приложение с push — вне scope.

Клиентское приложение: React и TypeScript

Пользовательский интерфейс — одностраничное приложение (SPA) на React 18.2. React выбран как наиболее распространённая библиотека для построения UI с компонентной декомпозицией, виртуальным DOM и развитой экосистемой (роутинг, тесты, календарь) [Источник №].

TypeScript (~5.6) добавляет статическую типизацию: интерфейсы ответов API в types/api.ts согласуются с Pydantic-схемами backend и снижают класс ошибок при рефакторинге.

Альтернативы.

Vue 3 — сопоставим по возможностям, но в учебной программе по направлению «Программная инженерия» чаще акцент на React; для единообразия стека с практиками курсовых работ выбран React.

Angular — полноценный фреймворк с жёсткой структурой; избыточен для приложения с ~10 экранами.

Чистый JavaScript без TypeScript — быстрее старт, но хуже сопровождаемость при росте числа компонентов.

Сборка и разработка frontend: Vite

Сборщик Vite 5 заменяет устаревший Create React App. Dev-сервер запускает модули через native ESM; горячая замена модулей (HMR) ускоряет итерации. Production-сборка выполняется через Rollup (команда npm run build → каталог dist).

Прокси в vite.config.ts перенаправляет запросы /api на backend при локальной разработке, что устраняет проблемы CORS на этапе отладки.

Альтернатива Next.js — SSR и маршрутизация на сервере; для приложения, где весь UI после авторизации работает в браузере с внешним API, классический SPA на Vite проще и прозрачнее.

Стили и маршрутизация

Tailwind CSS 3.4 — utility-first подход: классы в разметке задают отступы, сетку, адаптивные breakpoints (sm, md, lg). Это ускоряет вёрстку дашборда, таблиц и форм без написания большого объёма пользовательских CSS-файлов. PostCSS и Autoprefixer подключаются в цепочке сборки.

React Router 6 управляет маршрутами (/rooms, /my-bookings, /admin и др.) и вложенными layout; компоненты PrivateRoute и AdminRoute реализуют ограничение доступа на клиенте в дополнение к проверкам сервера.

Визуализация расписания: FullCalendar 6

Для отображения занятости комнаты подключены пакеты @fullcalendar/react, timegrid и daygrid (версия 6.1.x). Библиотека предоставляет виды «неделя» и «день» с почасовой сеткой, события загружаются из API бронирований и отображаются поверх формы создания брони.

Альтернатива — самописная сетка на div и CSS: трудоёмко, хуже доступность и перетаскивание. Альтернатива — платные UI-киты: не требуются.

Тестирование

Backend: Pytest 7.4+, pytest-asyncio, HTTP-клиент httpx для вызова ASGI-приложения без поднятия отдельного процесса. Тесты покрывают конфликты бронирований, политику отмены, коды ответов API. Для изоляции используется тестовая БД SQLite.

Frontend: Vitest 4 и среда happy-dom (или jsdom) для модульных тестов API-клиента и компонентов (HomePage, buildApiUrl). Выбор Vitest согласован с Vite — единая конфигурация, быстрый запуск.

Альтернатива Jest — возможна с React, но дублирует инструментарий при уже выбранном Vite.

Развёртывание и эксплуатация

Production-развёртывание выполняется на платформе Render по файлу render.yaml (Blueprint): managed PostgreSQL, Web Service для FastAPI (Uvicorn), Static Site для каталога dist после npm run build. Переменные окружения (DATABASE_URL, SECRET_KEY, TELEGRAM_BOT_TOKEN, VITE_API_URL) отделяют конфигурацию от кода (НФТ-9).

Альтернативы: VPS с Docker Compose — больше контроля, выше трудоёмкость администрирования; Heroku — сокращение бесплатного тарифа; Yandex Cloud / AWS — уместны в enterprise, для учебного прототипа Render достаточен.
