2.3. Проектирование REST API

Интерфейс взаимодействия клиента и сервера спроектирован в архитектурном стиле REST: ресурсы адресуются URL, операции задаются HTTP-методами, данные передаются в JSON. Схема согласована с моделью БД (п. 2.2) и реализуется роутерами FastAPI с префиксом /api.

Общие правила

Запросы и ответы — application/json. Для защищённых операций клиент передаёт JWT в заголовке Authorization: Bearer. Уровни доступа: публичный (без токена), авторизованный пользователь, администратор.

Типовые коды ответа: 200/201 — успех; 204 — успешное удаление без тела; 400 — нарушение бизнес-правил; 401 — нет или неверный токен; 403 — недостаточно прав; 404 — ресурс не найден; 409 — конфликт бронирования; 422 — ошибка валидации полей (Pydantic).

Входные и выходные структуры описаны Pydantic-схемами в каталоге schemas; спецификация OpenAPI генерируется автоматически и доступна по адресам /docs (Swagger UI) и /redoc.

Аутентификация и профиль

Группа маршрутов /api/auth и /api/users/me обслуживает регистрацию, подтверждение учётной записи, вход, восстановление пароля и профиль (табл. 2.3).

Таблица 2.3 – Маршруты аутентификации и профиля

| Метод | Путь | Доступ | Назначение |
| --- | --- | --- | --- |
| POST | /api/auth/register | Публичный | Регистрация, выдача ссылки на Telegram-бота |
| POST | /api/auth/verify-email | Публичный | Подтверждение кода, выдача JWT |
| POST | /api/auth/login | Публичный | Вход по email и паролю |
| POST | /api/auth/login-request | Публичный | Запрос кода для входа в Telegram |
| POST | /api/auth/login-verify | Публичный | Подтверждение кода входа, выдача JWT |
| POST | /api/auth/forgot-password | Публичный | Запрос кода сброса пароля |
| POST | /api/auth/reset-password | Публичный | Установка нового пароля по коду |
| GET | /api/auth/me | Авторизованный | Данные текущего пользователя |
| PATCH | /api/users/me | Авторизованный | Обновление профиля |

Вход и создание бронирований разрешены только после подтверждения учётной записи (проверка на сервере при login и в критичных операциях).

Переговорные комнаты и отзывы

Префикс /api/rooms. Просмотр списка и карточки комнаты — публичный; изменение справочника и загрузка фото — только администратор (табл. 2.4).

Таблица 2.4 – Маршруты комнат

| Метод | Путь | Доступ | Назначение |
| --- | --- | --- | --- |
| GET | /api/rooms | Публичный | Список с фильтрами (вместимость, поиск, регион, город и др.) |
| GET | /api/rooms/{id} | Публичный | Детали комнаты с фотографиями |
| POST | /api/rooms | Администратор | Создание комнаты |
| PATCH | /api/rooms/{id} | Администратор | Редактирование |
| DELETE | /api/rooms/{id} | Администратор | Удаление |
| POST | /api/rooms/{id}/photos | Администратор | Загрузка изображений (multipart) |
| DELETE | /api/rooms/{id}/photos/{photo_id} | Администратор | Удаление фото |
| GET | /api/rooms/{id}/reviews | Публичный | Список отзывов |
| POST | /api/rooms/{id}/reviews | Авторизованный | Добавление отзыва |

Бронирования

Префикс /api/bookings (табл. 2.5). Список броней по комнате и периоду открыт без авторизации — для отображения календаря занятости. Создание, просмотр «моих» броней, детали и отмена требуют JWT; отмена доступна только владельцу.

Таблица 2.5 – Маршруты бронирований

| Метод | Путь | Доступ | Назначение |
| --- | --- | --- | --- |
| GET | /api/bookings | Публичный | Брони комнаты за период (параметры room_id, from_time, to_time) |
| POST | /api/bookings | Авторизованный | Создание брони |
| GET | /api/bookings/me | Авторизованный | Список собственных бронирований |
| GET | /api/bookings/{id} | Авторизованный | Детали (владелец или администратор) |
| DELETE | /api/bookings/{id} | Авторизованный | Отмена (удаление записи) |

При создании и отмене сервер применяет бизнес-правила из п. 1.3 (длительность, буфер 15 минут, дедлайн отмены за 30 минут до начала).

Администрирование и служебные маршруты

Префикс /api/admin: список администраторов, назначение по email, снятие роли (с защитой «не менее одного админа» и запретом снять себя).

Webhook Telegram: GET и POST /api/telegram/webhook — приём обновлений от Bot API, привязка chat_id и доставка кодов.

Служебный GET /health возвращает статус доступности сервиса (мониторинг и деплой).

Итог

Спроектировано около тридцати HTTP-эндпоинтов, сгруппированных по предметным областям. Единый стиль URL, явное разграничение доступа и автодокументация упрощают разработку клиента (п. 2.4) и реализацию на этапе раздела 3.
