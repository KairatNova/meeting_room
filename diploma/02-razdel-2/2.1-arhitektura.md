2.1. Архитектура приложения

В разделе 1 сформулированы требования к веб-сервису бронирования переговорных. Ниже описана целевая архитектура: состав компонентов, их взаимодействие и принципы разбиения серверного кода. Детали хранения данных и контракт API раскрываются в п. 2.2 и 2.3.

Общая схема

Система построена по трёхуровневой клиент-серверной модели.

Уровень представления — одностраничное приложение (SPA) на React и TypeScript, выполняемое в браузере. Оно отвечает за навигацию, формы, календарь занятости и отображение данных, полученных с сервера.

Уровень приложения — HTTP-сервис на FastAPI (Python). Он принимает REST-запросы в формате JSON, проверяет права доступа, применяет бизнес-правила бронирования и обращается к базе данных.

Уровень данных — СУБД PostgreSQL для учётных записей, комнат, бронирований и служебных записей (коды подтверждения, отзывы). Фотографии комнат хранятся как файлы в каталоге uploads на сервере приложения и отдаются по URL /uploads.

Внешний сервис Telegram Bot API используется для доставки кодов регистрации, входа и сброса пароля; сервер принимает входящие сообщения через webhook.

Логическая схема взаимодействия приведена на рис. 2.1 (браузер — REST API — PostgreSQL; отдельно — канал к Telegram).

Клиентская часть

Фронтенд организован по функциональным каталогам. Страницы (pages) соответствуют маршрутам: каталог комнат, карточка с календарём, личные бронирования, профиль, админ-панель, аутентификация. Общая оболочка (Layout) задаёт шапку и навигацию. Модуль api инкапсулирует вызовы к backend; типы ответов описаны в types/api.ts. Состояние сессии пользователя хранится в AuthContext; защищённые маршруты реализованы обёртками PrivateRoute и AdminRoute.

При локальной разработке Vite проксирует запросы с префиксом /api на backend, что упрощает отладку без настройки CORS. В production статические файлы из каталога dist размещаются на хостинге, а VITE_API_URL указывает на публичный адрес API.

Серверная часть

Точка входа — main.py: создание приложения FastAPI, middleware CORS, подключение роутеров с префиксом /api, раздача загруженных файлов, эндпоинт /health для проверки доступности.

Внутренняя структура backend следует разделению ответственности (табл. 2.1).

Таблица 2.1 – Модули серверного приложения

| Каталог / модуль | Назначение |
| --- | --- |
| api/ | HTTP-маршруты: аутентификация, комнаты, бронирования, администрирование, webhook Telegram |
| models/ | ORM-модели SQLAlchemy, отображение таблиц БД |
| schemas/ | Pydantic-схемы валидации тел запросов и ответов |
| services/ | Бизнес-логика вне маршрутов: проверка конфликтов бронирований, работа с Telegram |
| core/ | Безопасность (хэш пароля, JWT), зависимости FastAPI (текущий пользователь, сессия БД) |
| config.py | Параметры из переменных окружения (БД, секреты, CORS, каталог загрузок) |

Роутеры не содержат сложной предметной логики: они вызывают сервисы и возвращают данные, прошедшие валидацию схем. Такой подход упрощает модульные тесты (например, отдельно для booking_conflict).

Поток обработки типового запроса

1. Браузер отправляет HTTPS-запрос с заголовком Authorization: Bearer при необходимости.
2. Middleware CORS и зависимость маршрута извлекают JWT и сессию БД.
3. Тело запроса проверяется Pydantic-схемой; при ошибке возвращается 422.
4. Выполняется операция (чтение или изменение через ORM / сервис).
5. Ответ сериализуется в JSON; для конфликта бронирования — код 409.

Развёртывание

В облаке (Render) размещаются три компонента: managed PostgreSQL, Web Service с Uvicorn для backend, Static Site для frontend. Миграции Alembic выполняются при старте backend. Конфигурация задаётся переменными окружения, без жёсткой привязки к конкретному хосту в исходном коде.

Итог

Выбранная архитектура отделяет интерфейс, бизнес-логику и данные, что соответствует требованиям п. 1.3 и облегчает сопровождение. Схема таблиц и связей описывается в п. 2.2; перечень и группировка REST-эндпоинтов — в п. 2.3; экраны и навигация клиента — в п. 2.4.
