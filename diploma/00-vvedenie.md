ВВЕДЕНИЕ

Управление переговорными комнатами — повседневная задача для организаций любого масштаба. Без автоматизации накапливаются системные сбои: двойные бронирования, непрозрачное расписание, потери рабочего времени на согласование конфликтов. Ручной учёт — таблицы Excel, бумажные журналы, устные договорённости — не даёт ни надёжности, ни единой картины занятости помещений.

На рынке представлены корпоративные и отраслевые решения. Крупные экосистемы (Microsoft Outlook Room Booking) и специализированные SaaS-сервисы (Robin Powered, Skedda) ориентированы на платные лицензии и зарубежную инфраструктуру. Отдельную группу составляют локальные IT-хабы и коворкинги, где бронирование встроено в сервис аренды помещений. Пример — Technopark в Бишкеке (https://technopark.kg/): круглосуточный технологический хаб площадью около 14 000 м², в котором резидентам и гостям доступны опенспейс, офисы, конференц- и ивент-залы, а также оборудованные переговорные комнаты; заявки на посещение и корпоративные услуги оформляются через сайт и личный кабинет площадки. Такой подход закрывает потребности конкретной локации, но не даёт организации произвольного внедрения своих правил бронирования, открытого API и независимого развёртывания — в отличие от разрабатываемого в работе веб-приложения.

Общий для многих зарубежных и площадочных решений пробел — слабая или отсутствующая интеграция с Telegram, мессенджером, широко используемым в Кыргызстане и странах СНГ. Для учебных заведений, небольших компаний и администраторов отдельного офиса разработка специализированной системы с гибкими бизнес-правилами и верификацией через бота остаётся практически оправданной альтернативой [Источник №].

Цель работы — разработать веб-приложение для автоматизации бронирования переговорных комнат с верификацией пользователей через Telegram и разграничением ролей (пользователь, администратор).

Для достижения поставленной цели поставлены следующие задачи:

1. Проанализировать предметную область и изучить существующие аналоги.
2. Сформулировать функциональные и нефункциональные требования к системе.
3. Обосновать выбор технологического стека.
4. Спроектировать архитектуру приложения и схему базы данных.
5. Реализовать серверную часть приложения (REST API, бизнес-логика).
6. Реализовать клиентскую часть приложения (пользовательский интерфейс).
7. Интегрировать верификацию аккаунтов через Telegram Bot API.
8. Разработать автоматические тесты и провести проверку системы.
9. Выполнить развёртывание приложения в облачной среде.

Объект исследования — процесс управления бронированием переговорных комнат в организации.

Предмет исследования — методы и технологии разработки веб-приложений для автоматизации процессов бронирования ресурсов.

В работе применялись: сравнительный анализ программных аналогов; объектно-ориентированное и реляционное моделирование данных; итеративная разработка; REST-архитектурный стиль проектирования API [Источник №]; модульное и интеграционное тестирование.

Разработанная система предназначена для эксплуатации в учебных заведениях, офисных центрах и коворкингах. Она развёрнута на облачной платформе Render, снабжена интерактивной документацией REST API (Swagger/OpenAPI) и проверена автоматическими тестами серверной и клиентской частей. Открытый технологический стек упрощает адаптацию правил бронирования под локальные требования.

Пояснительная записка состоит из введения, трёх разделов, заключения, списка использованных источников и приложений. В первом разделе рассматриваются предметная область, аналоги, требования и обоснование технологий. Второй раздел посвящён проектированию архитектуры, базы данных, API и пользовательского интерфейса. Третий раздел описывает реализацию, тестирование и развёртывание системы.
