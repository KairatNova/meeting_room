# Презентация проекта (для не-IT)

Этот файл помогает быстро и понятно рассказать о проекте на защите.

## 1. Что это за проект (20-30 секунд)

**Meeting Rooms** — веб-приложение для бронирования переговорных комнат.

Пользователь может:
- зарегистрироваться и подтвердить аккаунт через Telegram-бота,
- найти свободную комнату по фильтрам (дата, время, вместимость, удобства, локация),
- посмотреть занятость в календаре,
- забронировать комнату,
- открыть страницу своей брони и при необходимости отменить её,
- сохранять комнаты в избранное и видеть недавно просмотренные.

Администратор может:
- управлять комнатами (добавлять, редактировать, удалять),
- загружать фотографии комнат,
- назначать и снимать других администраторов.

## 2. Какую проблему решает

Без системы в компаниях часто возникают ошибки:
- два человека бронируют одну комнату на одно время,
- непонятно, какие комнаты свободны «прямо сейчас»,
- пользователи теряют свои бронирования и забывают детали.

Проект решает это автоматически:
- проверяет конфликты времени,
- показывает занятость в календаре,
- хранит бронирования в личном кабинете,
- добавляет понятные правила отмены и длительности.

## 3. Что показать на демо (по шагам)

1. **Регистрация и вход**
   - регистрация с Telegram (ник или номер),
   - переход в бота, получение кода, подтверждение,
   - вход в систему.

2. **Главная страница (дашборд)**
   - быстрый подбор свободных комнат,
   - блок ближайших бронирований,
   - избранные комнаты и недавно просмотренные.

3. **Страница комнаты**
   - фото, описание, параметры комнаты,
   - календарь занятости,
   - форма бронирования + обработка конфликта,
   - отзывы пользователей.

4. **Мои бронирования**
   - список будущих/прошедших,
   - переход на страницу конкретной брони,
   - отмена брони по правилам.

5. **Админ-панель**
   - CRUD комнат,
   - загрузка фото,
   - раздел управления администраторами.

## 4. Бизнес-логика (главное для оценки)

1) **Антиконфликт бронирований**
- система не даст занять уже занятый слот;
- дополнительно учтён буфер **15 минут** между бронями.

2) **Ограничения брони**
- минимум: **30 минут**,
- максимум: **6 часов**.

3) **Правила отмены**
- отмена доступна не позднее чем за **30 минут до начала**.

Итог: расписание получается реалистичным и без накладок.

## 5. Технологии (простыми словами)

- **Frontend:** React + TypeScript + Tailwind (интерфейс для пользователя).
- **Backend:** FastAPI + PostgreSQL (сервер, API, хранение данных).
- **FullCalendar:** визуальный календарь занятости.
- **JWT:** безопасная авторизация.
- **Telegram Bot API:** подтверждение аккаунта и коды.
- **Alembic:** миграции базы данных.
- **Vitest + Pytest:** автоматические тесты frontend и backend.

## 6. Что можно сказать в выводе

Проект готов как рабочий MVP:
- реализован полный путь пользователя: от регистрации до управления бронями;
- есть роли и админская часть;
- реализованы практические правила (буфер, лимиты, политика отмены);
- есть API-документация и тесты ключевой логики.
